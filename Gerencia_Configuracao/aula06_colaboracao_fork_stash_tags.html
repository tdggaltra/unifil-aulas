bash — fluxo fork completo
# 1. Fork pelo browser do GitHub (botão Fork) # 2. Clonar SEU fork $ git clone git@github.com:SEU_USER/projeto.git $ cd projeto # 3. Adicionar upstream (repositório original) $ git remote add upstream git@github.com:org/projeto.git $ git remote -v origin git@github.com:SEU_USER/projeto.git (fetch/push) upstream git@github.com:org/projeto.git (fetch/push) # 4. Criar branch e fazer sua contribuição $ git switch -c feature/minha-contribuicao $ git commit -am "feat: adiciona funcionalidade X" $ git push origin feature/minha-contribuicao # 5. Abrir PR no GitHub: fork → upstream (próxima aula!) # Sincronizar com upstream quando houver novidades $ git fetch upstream $ git merge upstream/main

🍴 Fork vs Clone — Qual a diferença?

  • Clone: cópia local de qualquer repo — você pode não ter permissão de push
  • Fork: cópia no seu GitHub — você tem push total e pode propor mudanças via PR
  • Fork preserva a conexão com o original (upstream) para sincronização futura

📄 Analogia: Fotocopiadora Acadêmica

Fork é como fotocopiar o artigo de outro professor para escrever suas anotações nas margens. O original fica intacto. Quando você terminar, entrega suas sugestões ao autor — que decide se incorpora ou não. Você nunca escreve no original diretamente.

🔁 Rebase Interativo — Esculpindo o Histórico

git rebase -i — a ferramenta de limpeza de commits antes do Pull Request

git rebase -i HEAD~N abre um editor com os últimos N commits — você decide o que fazer com cada um.
Comando Atalho O que faz Quando usar pick p Mantém o commit como está Padrão — commit está bom squash s Une este commit ao anterior, abrindo editor para nova mensagem Juntar vários WIP commits em um só significativo reword r Mantém o commit mas abre editor para editar a mensagem Corrigir mensagem com typo ou fora do padrão edit e Para no commit para você modificar os arquivos antes de continuar Dividir um commit grande em dois menores drop d Remove o commit completamente do histórico Eliminar commit de debug, senha acidental ou WIP inútil fixup f Como squash, mas descarta a mensagem do commit fundido Incorporar micro-fix sem poluir a mensagem final
bash — rebase -i na prática
# Histórico atual (confuso, com WIP commits) $ git log --oneline f3a1b2c wip b7e4d81 corrigindo mesmo a2c3f10 tentativa 2 9e1d4a7 feat: implementa autenticação JWT # Rebase interativo nos últimos 4 commits $ git rebase -i HEAD~4 # O editor abre com: (troque pick por s, s, s no segundo, terceiro e quarto) pick 9e1d4a7 feat: implementa autenticação JWT s a2c3f10 tentativa 2 s b7e4d81 corrigindo mesmo s f3a1b2c wip # Salve → editor abre para nova mensagem → escreva a mensagem final # Resultado: 1 commit limpo! $ git log --oneline c4f8d92 feat(auth): implementa autenticação JWT com validação completa # ⚠️ Após rebase, force push no branch pessoal (antes de abrir PR) $ git push --force-with-lease origin feature/autenticacao

✨ Squash Merge e Boas Práticas de PR

Mantendo o histórico do main limpo e linear

🎯 Três formas de integrar via PR

  • Merge commit: cria commit M com dois pais. Preserva histórico da feature. Ruidoso no log do main.
  • Squash and merge: comprime todos os commits da feature em um único no main. Log do main fica limpo.
  • Rebase and merge: replay linear dos commits da feature no topo do main. Sem merge commit.

🧹 Por que limpar antes do PR?

Seu branch de feature pode ter 20 commits de WIP, tentativas e correções. O revisor não precisa ver esse "making of" — ele quer ver o resultado. Squash ou rebase -i antes do PR é como entregar o relatório final, não seu caderno de rascunhos.

bash — squash e force-with-lease
# Squash manual antes de abrir PR # (alternativa ao rebase -i para caso simples) $ git switch main $ git merge --squash feature/login Squash commit -- not updating HEAD $ git commit -m "feat(auth): autenticação completa com JWT" # --force-with-lease: mais seguro que --force # Rejeita se alguém tiver feito push no meio tempo $ git push --force-with-lease origin feature/login To git@github.com:user/lab-git.git + b9e4d72...c4f8d92 feature/login -> feature/login (forced update) # Ver o commit final limpo $ git log --oneline -5 c4f8d92 feat(auth): autenticação completa com JWT g7h8f2d chore: configura gitignore

⚠️ Regra do force push

  • Nunca em main ou develop
  • Só em branches pessoais antes do merge
  • Avise o time quando usar em branch compartilhado
  • Prefira --force-with-lease ao --force

🗄️ git stash — Guardando Trabalho Inacabado

Troque de contexto sem perder uma linha de código

git stash empilha as mudanças não commitadas e limpa o Working Directory — como pausar um jogo sem perder o progresso.
git stash

Salva todas as mudanças não commitadas (tracked e staged) no stash e limpa o Working Directory.

git stash push -m "mensagem"

Stash com nome descritivo. Muito mais útil quando você tem vários stashes acumulados.

git stash list

Lista todos os stashes na pilha. stash@{0} é o mais recente, stash@{1} o anterior, etc.

git stash pop

Reaplica o stash mais recente e o remove da pilha. Use quando quiser restaurar e não precisar mais do stash.

git stash apply stash@{1}

Reaplica um stash específico sem removê-lo da pilha. Útil para aplicar em múltiplos branches.

git stash drop stash@{0}

Remove um stash específico sem reaplicar. Use stash clear para limpar todos de uma vez.

bash — cenário real de stash
# Trabalhando na feature quando chega o pedido urgente: # "O sistema de login está falhando em produção!" $ git status modified: login.c (work in progress...) # Não posso commitar — ainda não terminou. Faço stash! $ git stash push -m "WIP: refatorando login JWT" Saved working directory and index state On feature/login: WIP: refatorando login JWT # Agora o Working Directory está limpo — posso trocar de branch $ git switch main $ git flow hotfix start auth-fix $ git commit -am "fix: corrige token expirado em produção" $ git flow hotfix finish auth-fix # Pronto! Volto para a minha feature $ git switch feature/login $ git stash list stash@{0}: On feature/login: WIP: refatorando login JWT $ git stash pop On branch feature/login Changes not staged for commit: modified: login.c # ✅ Meu trabalho está de volta, exatamente onde parei!

🏷️ Tags — Marcando Versões no Tempo

Criando baselines imutáveis para releases e marcos do projeto

🏷️

Tag Anotada (annotated)

Recomendada para releases
Armazena: autor, data, mensagem e pode ser assinada com GPG. É um objeto Git completo, aparece em git tag -l e no GitHub como Release formal. Use sempre para versões de produção.
git tag -a v1.0.0 -m "Release 1.0.0 — versão estável"
🔖

Tag Leve (lightweight)

Para marcações internas
Apenas um ponteiro para um commit — sem metadados extras. Mais simples, sem mensagem. Use para marcos internos, pontos de referência durante desenvolvimento.
git tag v1.0.0-rc1
bash — criando e gerenciando tags
# Tag anotada — para release oficial $ git tag -a v1.0.0 -m "Release 1.0.0: primeira versão estável" # Tag em commit específico (não o atual) $ git tag -a v0.9.0 g7h8f2d -m "Beta release" # Listar todas as tags $ git tag -l v0.9.0 v1.0.0 # Ver detalhes de uma tag anotada $ git show v1.0.0 tag v1.0.0 Tagger: Tiago Galvão Date: Wed Jun 3 2026 Release 1.0.0: primeira versão estável commit g7h8f2d... # Publicar tags no GitHub (não são enviadas com git push normal!) $ git push origin v1.0.0 # uma específica $ git push origin --tags # todas de uma vez # Deletar tag local e remota $ git tag -d v1.0.0 $ git push origin --delete v1.0.0

🚀 Tags no GitHub = Releases

  • Tags anotadas → aba Releases do GitHub
  • GitHub gera automaticamente .zip e .tar.gz do código
  • Você pode adicionar changelog, binários e notas de release
  • GitHub Actions pode disparar deploy ao criar uma tag!

🧰 Outros Comandos Essenciais de Colaboração

cherry-pick · bisect · blame · shortlog

git cherry-pick abc1234

Aplica um commit específico (pelo hash) no branch atual, sem fazer merge do branch inteiro. Ideal para trazer um hotfix de develop para main sem levar outros commits.

git bisect start

Busca binária no histórico para encontrar qual commit introduziu um bug. Git divide o histórico ao meio e pergunta se está bom ou ruim — encontra o culpado em O(log n) passos.

git blame arquivo.c

Mostra quem modificou cada linha de um arquivo — com hash do commit, autor e data. Indispensável para entender decisões de código e rastrear introdução de bugs.

git shortlog -sn

Resumo de commits por autor em ordem decrescente. Ótimo para relatórios de contribuição ou verificar atividade da equipe no projeto.

bash — cherry-pick, blame e bisect
# Cherry-pick: trazer commit específico para main $ git switch main $ git cherry-pick abc1234 [main d3e4f5a] fix: corrige overflow na função soma 1 file changed, 2 insertions(+), 1 deletion(-) # Blame: ver quem escreveu cada linha $ git blame main.c g7h8f2d (Tiago Galvão 2026-04-29) #include <stdio.h> b9e4d72 (Ana Costa 2026-05-06) int soma(int a, int b){ b9e4d72 (Ana Costa 2026-05-06) return a + b; # Contribuições do time $ git shortlog -sn 18 Tiago Galvão 12 Ana Costa 7 Pedro Silva # Bisect para encontrar bug $ git bisect start $ git bisect bad # commit atual tem bug $ git bisect good v1.0.0 # v1.0.0 estava OK Bisecting: 4 revisions left to test after this (roughly 2 steps) # Git faz checkout automático no meio — teste, diga good/bad $ git bisect good # ou bad abc1234 is the first bad commit $ git bisect reset # volta ao HEAD

🧪 Laboratório — Prática Guiada

Fork · rebase -i · stash · tag — do zero ao fim

bash — roteiro completo aula 06
## PARTE 1: Rebase Interativo — Limpar histórico de commits $ cd ~/lab-git && git switch -c feature/refatoracao $ echo "// comentario" >> calc.c && git commit -am "wip" $ echo "// refatorado" >> calc.c && git commit -am "wip2" $ echo "// pronto" >> calc.c && git commit -am "feat: refatora funcoes" $ git log --oneline # veja os 3 commits "sujos" $ git rebase -i HEAD~3 # squash os 2 primeiros no terceiro # No editor: pick feat, squash wip2, squash wip $ git log --oneline # agora 1 commit limpo! ## PARTE 2: Stash — Interrupção e retomada $ echo "// trabalho novo" >> calc.c # começou algo novo $ git stash push -m "WIP: nova funcionalidade" $ git status # Working Directory limpo $ git switch main # pode trocar de branch sem medo $ git switch - # volta ao branch anterior $ git stash pop # restaura o trabalho ## PARTE 3: Tags — Marcar release $ git switch main $ git tag -a v1.1.0 -m "Release 1.1.0: calculadora completa" $ git tag -l # listar todas as tags $ git push origin --tags # publicar no GitHub # Verifique a aba Releases no GitHub!
