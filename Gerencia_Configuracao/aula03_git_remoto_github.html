☁️ Analogia: Google Drive para Código

🔐 Autenticação com Chaves SSH

Comunicação segura entre seu computador e o GitHub — sem senha toda hora

🗝️ O que é SSH Key Pair?

  • Chave privada (id_ed25519) — fica no SEU computador. Nunca compartilhe!
  • Chave pública (id_ed25519.pub) — vai para o GitHub. Pode compartilhar à vontade
  • Funciona como cadeado (público) + chave (privado)
  • Mais seguro que senha: imune a brute-force
  • Ed25519 = algoritmo moderno e recomendado

🏠 Analogia: Chave de Casa

Imagine que você dá um cadeado (chave pública) ao GitHub. Só quem tiver a chave física (privada) no bolso consegue abrir. Você pode dar cópias do cadeado para quantos servidores quiser — mas a chave fica apenas com você. 🔑

bash — gerando e registrando chave SSH
# PASSO 1: Gerar o par de chaves (pressione Enter em todas as perguntas) $ ssh-keygen -t ed25519 -C "tiago.galvao@unifil.br" Generating public/private ed25519 key pair. Enter file in which to save the key (~/.ssh/id_ed25519): [Enter] Enter passphrase (empty for no passphrase): [Enter] Your identification has been saved in ~/.ssh/id_ed25519 Your public key has been saved in ~/.ssh/id_ed25519.pub # PASSO 2: Adicionar ao agente SSH $ eval "$(ssh-agent -s)" Agent pid 12345 $ ssh-add ~/.ssh/id_ed25519 Identity added: /home/tiago/.ssh/id_ed25519 # PASSO 3: Copiar a chave PÚBLICA (cole no GitHub → Settings → SSH Keys) $ cat ~/.ssh/id_ed25519.pub ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... tiago.galvao@unifil.br # PASSO 4: Testar a conexão com o GitHub $ ssh -T git@github.com Hi tiago-galvao! You've successfully authenticated, but GitHub does not provide shell access.

📋 Onde registrar no GitHub

Settings → SSH and GPG keys → New SSH key → Cole o conteúdo de id_ed25519.pub → Título descritivo (ex: "Notebook UniFil 2026") → Add SSH key. Use a URL SSH dos repositórios (começa com git@github.com:), não a HTTPS.

🔗 git remote — Conectando Local ao Remoto

O elo entre seu repositório e o GitHub

git remote add origin git@github.com:user/repo.git

Registra o repositório remoto com o apelido origin. origin é a convenção — você pode usar qualquer nome, mas origin é universal.

git remote -v

Lista todos os remotos configurados e suas URLs (fetch e push separados). Verifique sempre antes de fazer push.

git remote remove origin

Remove o vínculo com o remoto. Útil ao trocar o repositório de URL ou de plataforma.

git remote set-url origin nova-url

Atualiza a URL do remote sem remover e re-adicionar. Ex: migrar de HTTPS para SSH.

bash — configurando o remote
# Dentro do repositório local (já com commits) $ git remote add origin git@github.com:tiago/lab-git.git # Verificar o que foi configurado $ git remote -v origin git@github.com:tiago/lab-git.git (fetch) origin git@github.com:tiago/lab-git.git (push) # Pode ter múltiplos remotos! $ git remote add upstream git@github.com:org/projeto-original.git $ git remote -v origin git@github.com:tiago/lab-git.git (fetch) origin git@github.com:tiago/lab-git.git (push) upstream git@github.com:org/projeto-original.git (fetch) upstream git@github.com:org/projeto-original.git (push)

📌 origin vs upstream

  • origin — seu fork ou repositório principal
  • upstream — o repositório original de onde fez fork
  • Padrão em projetos open-source para sincronizar com a fonte

📥 git clone — Baixando um Repositório Existente

A forma mais comum de começar a trabalhar num projeto

git clone copia o repositório remoto inteiro — histórico, branches, tags e configuração de remote — em um único comando.
git clone git@github.com:user/repo.git

Clona o repo na pasta repo/ no diretório atual. O remote origin já vem configurado automaticamente.

git clone URL meu-nome

Clona em pasta com nome personalizado. Útil para ter múltiplas cópias do mesmo projeto.

git clone --depth 1 URL

Clone raso — baixa apenas o último commit, sem todo o histórico. Muito mais rápido para repositórios enormes.

git clone -b develop URL

Clona o repositório e já faz checkout no branch develop em vez do padrão (main).

bash — clonando e explorando
# Clonar o repositório de exemplos da disciplina $ git clone git@github.com:tiago/gcv-exemplos.git Cloning into 'gcv-exemplos'... remote: Enumerating objects: 42, done. remote: Counting objects: 100% (42/42), done. remote: Compressing objects: 100% (28/28), done. Receiving objects: 100% (42/42), 15.2 KiB | 5.1 MiB/s, done. $ cd gcv-exemplos $ git remote -v origin git@github.com:tiago/gcv-exemplos.git (fetch) origin git@github.com:tiago/gcv-exemplos.git (push) # Ver histórico de quem contribuiu $ git log --oneline --graph * f9c1a8e docs: adiciona README completo * 3b7d2c1 feat: exemplos aula 02 * a1e4f9b chore: estrutura inicial

✅ clone vs init + remote add

  • Use clone quando o repositório já existe no GitHub
  • Use init + remote add quando você começa do zero localmente
  • Após clonar: já pode commitar e fazer push imediatamente

🚀 git push — Enviando Commits ao GitHub

Publicando seu trabalho para o mundo

git push origin main

Envia os commits locais do branch main para o remote origin. O mais usado no dia a dia.

git push -u origin main

O -u (ou --set-upstream) configura o tracking automático. Após isso, basta digitar git push nas próximas vezes.

git push --tags

Envia todas as tags locais para o remoto. Por padrão, git push não envia tags automaticamente.

git push --force ⚠️

Perigoso em repositórios compartilhados! Sobrescreve o histórico remoto. Use apenas em branches pessoais e jamais em main.

bash — primeiro push e push rotineiro
# Primeiro push: define o upstream (-u) $ git push -u origin main Enumerating objects: 6, done. Counting objects: 100% (6/6), done. Delta compression using up to 8 threads Compressing objects: 100% (4/4), done. Writing objects: 100% (6/6), 512 bytes | 512.00 KiB/s, done. Branch 'main' set up to track remote branch 'main' from 'origin'. # Pushes subsequentes — só isso! $ git push To git@github.com:tiago/lab-git.git b9e4d72..c7e1a09 main → main # Caso o push seja rejeitado (remoto mais novo) $ git push ! [rejected] main → main (fetch first) hint: Updates were rejected because the remote contains work # Solução: git pull antes de fazer push

📬 Push = "Postar uma carta"

Seus commits são cartas escritas. git push as leva ao correio (GitHub). Antes de postar, confirme que você tem as últimas cartas recebidas — senão o carteiro recusa a entrega (rejected).

📥 git pull e git fetch — Recebendo Atualizações

Sincronizando com o que o time fez enquanto você estava trabalhando

📦

git fetch

Baixa os commits do remoto para o repositório local
mas NÃO aplica no Working Directory.
Você inspeciona antes de integrar.

Remote → .git/refs
VS
⚡

git pull

Equivale a fetch + merge em um comando.
Baixa E aplica as mudanças
imediatamente no branch atual.

Remote → .git + Working Dir
git pull

O mais usado. Baixa e integra commits do branch rastreado. Se houver conflitos, o Git sinaliza para resolução manual.

git pull --rebase

Em vez de merge, aplica seus commits locais em cima dos remotos. Mantém histórico linear. Preferido por muitas equipes.

git fetch origin

Seguro: baixa tudo sem alterar nada. Depois inspecione com git log origin/main antes de integrar.

bash — pull, fetch e fluxo seguro
# Fluxo rotineiro do dia a dia $ git pull remote: Enumerating objects: 3, done. Updating b9e4d72..f1c3a90 Fast-forward README.md | 4 +++- 1 file changed, 3 insertions(+), 1 deletion(-) # Fluxo seguro com fetch + inspeção $ git fetch origin $ git log HEAD..origin/main --oneline f1c3a90 docs: atualiza README com exemplos # Parece OK! Vou integrar: $ git merge origin/main Updating b9e4d72..f1c3a90 # Pull com rebase (histórico linear) $ git pull --rebase origin main Successfully rebased and updated refs/heads/main.

💡 Regra de Ouro do trabalho em equipe

Sempre faça git pull antes de começar a trabalhar e antes de fazer push. Isso evita a maioria dos conflitos e rejeições de push.

🗺️ URLs: SSH vs HTTPS

  • SSH: git@github.com:user/repo.git — autenticação por chave, sem senha
  • HTTPS: https://github.com/user/repo.git — pede token ou senha toda vez
  • Use sempre SSH no dia a dia profissional

⚡ Atalhos do GitHub CLI

  • gh repo clone user/repo — clone simplificado
  • gh pr create — cria Pull Request pelo terminal
  • Instale: cli.github.com

📄 README.md — A Vitrine do seu Projeto

Documentação que faz a diferença para recrutadores e colaboradores

O README é a primeira coisa que qualquer pessoa vê ao abrir seu repositório. Um README bem feito vale mais que mil palavras no currículo.

📋 Estrutura de um README Completo

  • Título e badges (build passing, versão, licença)
  • Descrição — o que o projeto faz em 2 linhas
  • Pré-requisitos — o que precisa instalado
  • Instalação — passo a passo com comandos
  • Como usar — exemplos de uso com código
  • Contribuição — como colaborar (pull requests)
  • Licença — MIT, Apache, GPL…

4. 💬 Dissertativa — Cenário real:

Você e mais dois colegas estão desenvolvendo um projeto. Você faz git push e recebe o erro "rejected — fetch first". Explique por que esse erro ocorre e descreva passo a passo o que você deve fazer para resolver sem perder seus commits.

💡 Gabarito: O erro ocorre porque o remoto tem commits que você não tem localmente — um colega fez push enquanto você trabalhava. O Git rejeita para evitar perda de dados. Solução: (1) git pull para baixar e integrar os commits do colega, (2) resolver conflitos se houver, (3) git push novamente. Opcionalmente: git pull --rebase mantém histórico linear.
