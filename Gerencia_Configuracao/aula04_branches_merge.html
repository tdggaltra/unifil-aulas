🎯 Objetivo desta aula

Criar e navegar entre branches com fluência, executar os dois tipos de merge, resolver conflitos sem perder o trabalho e entender quando preferir rebase ao merge.

🎭 Pré-aquecimento: por que branches importam?

Imagine 4 devs trabalhando no mesmo arquivo de código simultaneamente — direto no main. Cada git push poderia apagar o trabalho do outro. Branches resolvem isso: cada dev trabalha em sua própria "linha paralela" do tempo e só une ao principal quando o trabalho está pronto e revisado.

🌿 O que é um Branch?

Linhas paralelas de desenvolvimento — leves, rápidas e poderosas

Um branch é simplesmente um ponteiro móvel para um commit. Criar um branch custa microsegundos e ocupa apenas 41 bytes em disco.

🌿 Branch main

  • Branch padrão criado com git init
  • Representa o código estável e pronto
  • Antes chamado de master — renomeado por boas práticas
  • Protegido em produção: ninguém commita direto

🔵 O que é HEAD?

  • Ponteiro especial que indica "onde você está agora"
  • Normalmente aponta para o branch ativo
  • Ao trocar de branch, HEAD se move
  • Detached HEAD: aponta direto para um commit

⚡ Por que branches são tão rápidos?

  • Branches são só arquivos de texto (41 bytes!)
  • Não copiam arquivos — apenas apontam para commits
  • SVN levava minutos para ramificar
  • Git: milissegundos — por isso o GitFlow funciona

🛠️ Criando e Navegando entre Branches

branch · checkout · switch — o arsenal completo

git branch feature/login

Cria o branch mas não troca para ele. Você continua no branch atual.

git switch feature/login

Troca para o branch especificado. Comando moderno — substitui o git checkout para navegação.

git switch -c feature/login

Cria e já troca para o novo branch em um único passo. O mais usado no dia a dia.

git branch

Lista todos os branches locais. O branch atual aparece marcado com *. Use -a para ver os remotos também.

git branch -d feature/login

Deleta o branch após o merge (seguro). Use -D para forçar mesmo sem merge.

bash — criando e navegando branches
# Ver branches existentes $ git branch * main # Criar e trocar em um passo $ git switch -c feature/login Switched to a new branch 'feature/login' # Verificar onde estamos $ git branch * feature/login main # Fazer commits na feature $ git commit -am "feat(auth): cria formulário de login" $ git commit -am "feat(auth): valida email e senha" # Voltar ao main (commits da feature ficam no branch!) $ git switch main Switched to branch 'main' # Listar branches com último commit $ git branch -v feature/login b3c1a90 feat(auth): valida email e senha * main g7h8f2d chore: configura gitignore

📌 Convenção de nomes de branch

  • feature/nome-da-feature — nova funcionalidade
  • fix/descricao-do-bug — correção de bug
  • hotfix/urgente — correção emergencial
  • docs/atualiza-readme — documentação

⚡ Merge Fast-Forward

O merge mais simples — quando não há desvio no histórico

⚡

Fast-Forward Merge

Condição: o branch de destino (main) não teve novos commits desde que o feature foi criado.

O que acontece: o ponteiro do main simplesmente "avança" para o último commit da feature. Histórico limpo, linear.

Resultado: sem commit de merge extra — como se tudo tivesse sido feito diretamente no main.

bash — fast-forward merge
# Voltar ao main e fazer o merge $ git switch main $ git merge feature/login Updating g7h8f2d..b3c1a90 Fast-forward login.c | 42 +++++++++++++++++++ 1 file changed, 42 insertions(+) # Histórico fica linear! $ git log --oneline --graph * b3c1a90 feat(auth): valida email e senha * a8d3f10 feat(auth): cria formulário de login * g7h8f2d chore: configura gitignore # Deletar o branch após o merge $ git branch -d feature/login Deleted branch feature/login (was b3c1a90).

🔀 Merge 3-Way (True Merge)

Quando ambos os branches evoluíram — o Git une os dois mundos

🔀

3-Way Merge

Condição: ambos os branches têm commits novos desde a divergência.

O que acontece: Git usa 3 pontos (ancestral comum + ponta de cada branch) para criar um novo "merge commit" M com dois pais.

Resultado: histórico preserva a linha do tempo real — visível no git log --graph.

bash — 3-way merge
$ git switch main $ git merge feature/login Merge made by the 'ort' strategy. login.c | 42 +++++++++++++++ 1 file changed, 42 insertions(+) # Histórico mostra ramificação $ git log --oneline --graph * f7a2c3e Merge branch 'feature/login' |\ | * e9d1b4a feat(auth): valida email e senha | * a8d3f10 feat(auth): cria formulário * | c3f2a81 fix: corrige bug no dashboard |/ * g7h8f2d chore: configura gitignore

⚔️ Resolvendo Conflitos de Merge

O que acontece quando dois branches editam a mesma linha

Conflito não é erro — é o Git pedindo sua opinião sobre qual versão manter quando ele não consegue decidir sozinho.
bash — detectando conflito
$ git merge feature/login Auto-merging main.c CONFLICT (content): Merge conflict in main.c Automatic merge failed; fix conflicts and then commit the result. $ git status Both modified: main.c You have unmerged paths. (fix conflicts and run "git commit") (use "git merge --abort" to abort the merge)
<<<<<<< HEAD (main) void autenticar(char *usuario) { printf("Login: %s\n", usuario); } ======= int autenticar(char *usuario, char *senha) { return strcmp(senha, obterSenha(usuario)) == 0; } >>>>>>> feature/login
<<<<<<< HEAD SUA versão (HEAD/main) VERSÃO do feature

🔴 Passo a Passo para Resolver

  • 1. git status — listar arquivos em conflito
  • 2. Abrir o arquivo no VS Code (marcadores coloridos)
  • 3. Editar — manter o que faz sentido (pode ser ambos!)
  • 4. Remover todos os marcadores <<<, ===, >>>
  • 5. git add arquivo_resolvido.c
  • 6. git commit — o Git gera a mensagem de merge
bash — após resolver o conflito
# Após editar main.c e resolver o conflito $ git add main.c $ git status All conflicts fixed but you are still merging. (use "git commit" to conclude merge) $ git commit [main f7a2c3e] Merge branch 'feature/login' # Para abortar e voltar ao estado anterior $ git merge --abort

💡 VS Code facilita conflitos

O VS Code exibe botões "Accept Current", "Accept Incoming" e "Accept Both" acima de cada conflito. Use a extensão GitLens para ainda mais controle visual.

🔁 Rebase — Histórico Linear e Elegante

A alternativa ao merge que mantém o histórico limpo

git rebase main

Dentro do branch feature: reaplica os commits da feature em cima do topo atual de main. Cria novos hashes (D', E').

git rebase -i HEAD~3

Rebase interativo: permite reordenar, editar mensagens, unir (squash) ou remover commits dos últimos 3. Poderoso para limpeza antes do PR.

git rebase --abort

Cancela o rebase em andamento e volta ao estado anterior. Equivalente ao merge --abort.

⚖️ Merge vs Rebase — Quando usar cada um?

Cenário Merge Rebase
Integrar feature em main ✅ ✅
Manter histórico real ✅ ❌
Histórico linear/limpo ❌ ✅
Branch compartilhado (remoto) ✅ ⚠️ cuidado!

🚂 Rebase = "Trilho novo"

Mergear é juntar dois trens numa bifurcação. Rebase é pegar todos os vagões da feature e colocá-los no fim do trilho do main — como se sempre tivessem sido construídos lá.

🧪 Laboratório — Prática Guiada

Branches, merge e conflito do zero ao fim

bash — roteiro completo aula 04
## PARTE 1: Fast-Forward Merge $ cd ~/lab-git $ git switch -c feature/calculadora $ cat > calc.c << 'EOF' #include<stdio.h> int soma(int a,int b){return a+b;} int main(){printf("%d\n",soma(3,4));return 0;} EOF $ git add calc.c && git commit -m "feat: adiciona função soma" $ git switch main $ git merge feature/calculadora # → Fast-forward! main avança para o commit da feature $ git log --oneline --graph $ git branch -d feature/calculadora ## PARTE 2: 3-Way Merge com conflito $ git switch -c feature/subtracao $ echo "int sub(int a,int b){return a-b;}" >> calc.c $ git commit -am "feat: adiciona função subtração" $ git switch main $ echo "int mult(int a,int b){return a*b;}" >> calc.c $ git commit -am "feat: adiciona função multiplicação" $ git merge feature/subtracao # → Ambos editaram calc.c: CONFLITO! # Abra o arquivo, resolva os marcadores e salve $ git add calc.c && git commit $ git log --oneline --graph --all ## PARTE 3: Push dos branches para o GitHub $ git push

✅ Desafio A

Crie uma branch feature/divisao, adicione função div(a,b) com proteção contra divisão por zero, faça o merge e observe o tipo de merge com git log --graph.

✅ Desafio B — Rebase

Em vez de merge, use git rebase main no branch feature/divisao e depois faça fast-forward merge. Compare o git log --graph com o resultado anterior.

🔥 Desafio C (Bônus)

Use git rebase -i HEAD~3 para combinar (squash) os 3 últimos commits em um único commit bem descrito. Confirme com git log --oneline.

4. 💬 Dissertativa — Análise comparativa:

💡 Gabarito orientativo — Merge: (1) Preserva histórico real de como o trabalho aconteceu; (2) Seguro para branches compartilhados/remotos. Rebase: (1) Histórico linear e mais fácil de ler com git log; (2) Elimina merge commits desnecessários. Uso: merge para branches compartilhados em equipe; rebase para limpeza de branches locais antes de abrir um Pull Request.
