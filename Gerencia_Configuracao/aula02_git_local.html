⚙️ Configuração Global (uma vez só)

  • Nome do autor: aparece em todo commit
  • E-mail: vinculado à conta GitHub
  • Editor padrão: usado para mensagens longas
  • Configuração fica em ~/.gitconfig
bash — configuração global do git
# Identidade — substitua pelos seus dados $ git config --global user.name "Tiago Galvão" $ git config --global user.email "tiago.galvao@unifil.br" # Editor padrão (VS Code) $ git config --global core.editor "code --wait" # Branch principal chamada 'main' (padrão moderno) $ git config --global init.defaultBranch main # Verificar tudo que foi configurado $ git config --global --list user.name=Tiago Galvão user.email=tiago.galvao@unifil.br core.editor=code --wait init.defaultbranch=main

🪪 Por que configurar nome e e-mail?

Git é um sistema colaborativo. Cada commit fica gravado com o autor. Assim, quando você olhar o histórico de um projeto com 10 devs, saberá exatamente quem escreveu cada linha — como uma assinatura em contrato. Sem essa config, o Git se recusa a fazer commits.

🗺️ As 3 Áreas do Git

O conceito mais importante para entender como o Git pensa

📁
Working Directory
Seu diretório local. Arquivos que você está editando agora. O Git enxerga as mudanças, mas ainda não as rastreia.
arquivos modificados arquivos novos
→
🎯
Staging Area (Index)
Área de preparação. Você escolhe exatamente o que vai para o próximo commit. Como preparar uma caixa antes de lacrar.
git add arquivo mudanças selecionadas
→
🏛️

O banco de dados do Git. Histórico permanente e completo de todos os commits. Fica na pasta oculta .git/

☁️
Remote (GitHub)
Repositório remoto compartilhado. Backup e colaboração. Próxima aula!

📦 Analogia: Preparando um Pacote para Entrega

Working Directory = sua mesa cheia de itens. Staging = a caixa que você está montando — você escolhe o que coloca nela. Repository = o pacote lacrado com etiqueta de rastreio — permanente, imutável. O git add coloca o item na caixa. O git commit lacra e etiqueta.

untracked

Arquivo novo que o Git ainda não conhece

modified

Arquivo já rastreado, com mudanças não staged

staged

Mudança na staging area, pronta para commit

🏁 git init e git status

Criando um repositório e lendo o que o Git está vendo

bash — criando o primeiro repositório
# 1. Criar pasta e entrar nela $ mkdir meu-projeto && cd meu-projeto # 2. Inicializar o repositório Git $ git init Initialized empty Git repository in /home/tiago/meu-projeto/.git/ # 3. Verificar o estado atual $ git status On branch main No commits yet nothing to commit (create/copy files and use "git add" to track) # 4. Criar um arquivo e verificar novamente $ echo "# Meu Projeto" > README.md $ git status On branch main No commits yet Untracked files: (use "git add <file>..." to include in what will be committed) README.md nothing added to commit but untracked files present
git init

Cria o diretório oculto .git/ com toda a estrutura interna do Git. Deve ser executado uma vez, na raiz do projeto. Não apague jamais a pasta .git/ — ela É o repositório.

git status

Mostra o estado atual de cada arquivo. É o seu "GPS do repositório" — use sempre antes de add e commit para saber exatamente o que está acontecendo. Execute sem medo, ele não faz nada.

💡 O que tem dentro de .git/?

.git/HEAD — aponta para o branch atual  ·  .git/objects/ — todos os commits, árvores e blobs  ·  .git/config — configurações locais do repositório  ·  .git/refs/ — branches e tags. Nunca edite manualmente!

🎯 git add — A Área de Staging

Escolhendo com precisão o que vai para o próximo commit

git add arquivo.txt

Adiciona um arquivo específico ao staging. Use quando quer commitar apenas parte das mudanças.

git add .

Adiciona tudo do diretório atual ao staging — novos arquivos, modificações e exclusões.

git add -p

Modo interativo: mostra cada trecho modificado e pergunta se você quer staged-lo. Perfeito para commits cirúrgicos.

git restore --staged arquivo

Remove um arquivo da staging area sem desfazer as mudanças no disco. "Tira da caixa antes de lacrar."

bash — fluxo de git add
# Criamos 3 arquivos $ touch main.c utils.c README.md # Adicionamos apenas main.c e README $ git add main.c README.md $ git status Changes to be committed: new file: README.md new file: main.c Untracked files: utils.c # Adicionamos o restante $ git add utils.c $ git status Changes to be committed: new file: README.md new file: main.c new file: utils.c # Ops, utils.c não deveria ir ainda... $ git restore --staged utils.c

🛒 Por que staging existe? Por que não commitar direto?

Imagine que você refatorou 5 arquivos mas quer criar 2 commits separados: um para a correção de bug, outro para a nova feature. O staging permite exatamente isso — você seleciona apenas os arquivos do bugfix, commita, depois seleciona os da feature e commita novamente. Commits menores e focados = histórico mais legível = rollbacks mais precisos.

💾 git commit — Registrando Mudanças

Criando snapshots permanentes com mensagens que importam

git commit -m "mensagem"

Commit rápido com mensagem inline. Suficiente para a maioria dos casos.

git commit --amend

Modifica o último commit — corrige a mensagem ou adiciona arquivos esquecidos. ⚠️ Só use antes do push!

git commit -am "mensagem"

Atalho que faz add + commit de arquivos já rastreados. Não inclui arquivos novos (untracked).

bash — primeiro commit
$ git commit -m "feat: adiciona estrutura inicial do projeto" [main (root-commit) a3f2c1d] feat: adiciona estrutura inicial 3 files changed, 0 insertions(+), 0 deletions(-) create mode 100644 README.md create mode 100644 main.c create mode 100644 utils.c # Edita main.c e commita novamente $ git commit -am "feat: implementa função main com menu" [main b9e4d72] feat: implementa função main com menu 1 file changed, 15 insertions(+)

📝 Conventional Commits — O padrão profissional de mensagens

feat: nova funcionalidade
→ incrementa MINOR
fix: correção de bug
→ incrementa PATCH
refactor: sem mudança de comportamento
docs: documentação
test: adiciona ou corrige testes
chore: tarefas de build, config

Padrão completo: tipo(escopo): descrição → ex: feat(auth): adiciona login com Google

🔍 git log e git diff — Inspecionando o Histórico

Entendendo o passado para tomar decisões no presente

git log

Exibe histórico completo — hash, autor, data, mensagem de cada commit.

git log --oneline

Uma linha por commit. Hash curto + mensagem. Muito mais legível para projetos grandes.

git log --oneline --graph --all

Exibe gráfico ASCII do histórico com todos os branches. Essencial para entender merges.

git diff

Mostra o que mudou nos arquivos ainda não staged — linha a linha, com +verde (adição) e -vermelho (remoção).

bash — git log e git diff
$ git log --oneline b9e4d72 feat: implementa função main com menu a3f2c1d feat: adiciona estrutura inicial do projeto $ git log --oneline --graph --all * b9e4d72 feat: implementa função main com menu * a3f2c1d feat: adiciona estrutura inicial do projeto # Ver o que mudou em main.c antes de staged $ git diff main.c diff --git a/main.c b/main.c --- a/main.c +++ b/main.c @@ -1,4 +1,7 @@ #include <stdio.h> + +void exibirMenu() { + printf("1. Novo\n2. Sair\n"); +} # Ver mudanças já na staging area $ git diff --staged
git show b9e4d72

Exibe o diff completo de um commit específico pelo hash.

git restore arquivo

Descarta mudanças no Working Dir — volta ao último commit. ⚠️ Irreversível!

git log --author="Tiago"

Filtra commits por autor. Útil para auditar contribuições em equipe.

🙈 .gitignore — O que NÃO Versionar

Mantendo o repositório limpo e seguro

O .gitignore lista padrões de arquivos que o Git deve ignorar completamente — como se não existissem.

🚫 O que NUNCA versionar

  • Credenciais e segredos: .env, senhas, API keys, tokens
  • Dependências: node_modules/, .venv/, packages baixados
  • Arquivos de build: .exe, .o, .class, dist/
  • Cache e temporários: .DS_Store, Thumbs.db, *.tmp
  • IDEs/editores: .idea/, .vscode/ (settings pessoais)
  • Logs: *.log, npm-debug.log
.gitignore — projeto C / Node.js
# Compilados C *.o *.exe *.out # Node.js node_modules/ npm-debug.log # Credenciais — NUNCA commite isso! .env *.key secrets.json # Sistema Operacional .DS_Store # macOS Thumbs.db # Windows # IDEs .vscode/settings.json .idea/
cat .gitignore

Exibe o conteúdo do .gitignore atual. O arquivo fica na raiz do projeto e é versionado junto com o código.

git check-ignore -v arquivo

Verifica se um arquivo está sendo ignorado e qual regra do .gitignore é responsável. Ótimo para debugar.

🌐 gitignore.io — Gerando .gitignore automaticamente

Acesse gitignore.io (ou git.io/gitignore) e informe sua linguagem/IDE. O site gera um .gitignore completo e atualizado. No GitHub, ao criar um repositório, você já pode escolher um template de .gitignore automaticamente!

🧪 Laboratório — Prática Guiada

Execute cada passo no terminal e observe os resultados

bash — roteiro completo da aula 02
## PARTE 1 — Configurar e criar repositório $ git config --global user.name "Seu Nome" $ git config --global user.email "seu@email.com" $ mkdir lab-git && cd lab-git && git init ## PARTE 2 — Criar .gitignore e README $ echo -e "*.exe\n*.o\n.env" > .gitignore $ echo "# Lab Git — Aula 02" > README.md $ git add . $ git commit -m "chore: configura gitignore e README" ## PARTE 3 — Criar código C e commitar $ cat > main.c << 'EOF' #include <stdio.h> int main() { printf("Olá, Git!\n"); return 0; } EOF $ git add main.c $ git status # observe o arquivo em verde (staged) $ git commit -m "feat: adiciona programa hello world em C" ## PARTE 4 — Modificar e inspecionar $ echo 'void saudacao(){printf("Bem-vindo!\n");}' >> main.c $ git diff # veja a linha adicionada em verde $ git commit -am "feat: adiciona função saudacao" ## PARTE 5 — Ver histórico $ git log --oneline --graph * c7e1a09 feat: adiciona função saudacao * 4b2f831 feat: adiciona programa hello world em C * e5d3c12 chore: configura gitignore e README

✅ Desafio B

2. Um desenvolvedor executa git diff e não vê nenhuma saída, mas sabe que fez mudanças. O que mais provavelmente aconteceu?

4. 💬 Dissertativa — Explique em 2-3 linhas:

Por que é uma má prática commitar o arquivo .env (com credenciais) no repositório Git? Quais são as consequências reais no mercado profissional?

💡 Gabarito: Uma vez que o .env é commitado, suas credenciais ficam permanentemente no histórico do Git — mesmo após deletar o arquivo. Se o repositório for público (GitHub), bots varrem commits em segundos e podem usar as chaves de API ou acessar bancos de dados. Empresas como AWS, GitHub e Google têm sistemas que detectam e notificam vazamentos — mas o dano pode já ter ocorrido. O .gitignore deve sempre listar .env antes do primeiro commit.
