Soa familiar? Agora imagine isso com código, em equipe, com 50 arquivos. 🤯

❌ Problema 1 — Histórico perdido

Você não sabe o que mudou entre uma versão e outra, nem quem fez a mudança ou por quê.

❌ Problema 2 — Trabalho em equipe

Dois devs editam o mesmo arquivo. Um sobrescreve o trabalho do outro. Horas de código perdidas para sempre.

❌ Problema 3 — Sem retorno seguro

Você fez uma mudança que quebrou tudo. Sem controle de versão, não tem como voltar ao estado anterior que funcionava.

Controle de versão é o histórico médico do seu código — cada consulta, diagnóstico e tratamento registrado com data e responsável. 🏥

⚙️ Gerência de Configuração de Software (GCS)

Conceito, definição e disciplina de engenharia

GCS é a disciplina de Engenharia de Software que identifica, organiza e controla as modificações no software ao longo de todo o ciclo de vida do produto.

📌 O que a GCS controla?

  • Código-fonte — todos os arquivos .c, .java, .py…
  • Documentação — requisitos, diagramas, manuais
  • Arquivos de configuração — .env, .yaml, .json
  • Scripts de build e deploy — Makefile, Dockerfile
  • Testes automatizados — arquivos de teste e fixtures
  • Assets — imagens, fontes, ícones do projeto

🎯 Objetivos da GCS

  • Garantir rastreabilidade de todas as mudanças
  • Permitir reproduzir qualquer versão do sistema
  • Evitar conflitos em trabalho paralelo em equipes
  • Facilitar a auditoria e o controle de qualidade
  • Suportar entrega e implantação controladas
  • Possibilitar rollback seguro em caso de falha

🏗️ Analogia: GCS é como a Engenharia Civil

Na construção de um prédio, toda planta, especificação de material e ordem de serviço é documentada, versionada e aprovada antes de ir para a obra. Nenhuma alteração estrutural é feita sem registro. A GCS faz o mesmo com software: toda mudança passa por um processo controlado — com quem, quando, o que e por quê.

📚 Norma IEEE 828 — Padrão de GCS

O IEEE 828 define que um plano de GCS deve cobrir: identificação de configuração, controle de configuração, auditoria de configuração e relatório de status — os quatro pilares que o Git implementa nativamente.

🔑 Itens de Configuração e Conceitos Fundamentais

O vocabulário essencial da GCS

Conceito Definição Exemplo no Git 🗂️ Item de Configuração (IC) Qualquer artefato que está sob gerência de configuração — código, doc, config, teste Qualquer arquivo rastreado: arquivo.c, .env, README.md 📍 Baseline Fotografia do estado do sistema em um ponto específico — aprovada e imutável Uma tag no Git: git tag v1.0.0 🚀 Release Versão do software entregue ao cliente ou ao ambiente de produção Branch release/1.0 + tag v1.0.0 🔨 Build Processo de compilar/empacotar o código em um artefato executável Pipeline CI no GitHub Actions que compila e testa 🔄 Revisão / Commit Unidade atômica de mudança — conjunto de alterações salvas no repositório git commit -m "feat: adiciona login" 📜 Log de Mudanças Histórico completo de quem mudou o quê, quando e por quê git log --oneline --graph

📸 Baseline = "Checkpoint" de um Jogo

Em jogos, você salva em um checkpoint antes de uma batalha difícil. Se morrer, volta ao ponto salvo — não ao início do jogo. Na GCS, uma baseline é esse checkpoint: uma versão estável e aprovada à qual você pode sempre retornar, independentemente de quantas mudanças foram feitas depois.

🏷️ Versionamento Semântico (SemVer)

A linguagem universal dos números de versão

MAJOR . MINOR . PATCH  →  2 . 14 . 3

🔴 MAJOR — Versão Principal

Incrementa quando há mudanças incompatíveis com versões anteriores (breaking change).

Ex: 1.x.x → 2.0.0

React 17 → React 18

🟡 MINOR — Nova Funcionalidade

Incrementa quando se adiciona nova funcionalidade mantendo compatibilidade.

Ex: 1.3.x → 1.4.0

Node.js 20.0 → 20.1

🟢 PATCH — Correção de Bug

Incrementa quando há correções de falhas sem alterar a interface pública.

Ex: 1.3.2 → 1.3.3

Hotfix de segurança

📝 Sufixos Especiais

  • 1.0.0-alpha — versão alfa, instável, uso interno
  • 1.0.0-beta.1 — versão beta, testes externos
  • 1.0.0-rc.1 — Release Candidate, candidato a lançamento
  • 1.0.0 — versão estável, produção

💡 Regra de Ouro do SemVer

Se você conserta um bug → PATCH. Se você adiciona algo novo mas não quebra nada → MINOR. Se você muda algo que vai quebrar quem usa sua biblioteca → MAJOR. Uma vez publicada, uma versão nunca pode ser alterada — crie uma nova.

🕐 Evolução dos Sistemas de Controle de Versão

De arquivos com "_v2_FINAL" ao Git distribuído

Pré-1972
Sem VCS

Cópias manuais · pastas datadas · risco total de perda

1972–1985
SCCS / RCS

Primeiro VCS real · local · um arquivo por vez · sem rede

1990–2000
CVS

Concurrent Versions System · centralizado · primeiro VCS em rede · branches lentas

2000–2005
SVN

Subversion · melhorou o CVS · commits atômicos · ainda centralizado · usado até hoje em legado

2005 →
Git

Linus Torvalds · distribuído · branches baratas · rápido · padrão absoluto do mercado

2008 →
GitHub · GitLab

Plataformas remotas · Pull Requests · Actions · CI/CD · code review social

⚡ Por que o Git surgiu em 2005?

Linus Torvalds criou o Git em apenas 10 dias para gerenciar o kernel do Linux após uma crise com o BitKeeper (VCS proprietário que revogou a licença gratuita). Ele precisava de algo rápido, distribuído e confiável para coordenar milhares de desenvolvedores ao redor do mundo. O resultado foi o VCS que hoje é usado por mais de 94% dos desenvolvedores no mundo.

🏛️ Centralizado (CVCS) × Distribuído (DVCS)

A diferença fundamental de arquitetura que mudou o desenvolvimento

🏢

CVCS — Centralizado

CVS · SVN · Perforce

  • 📡 Um servidor central guarda todo o histórico
  • 💻 Desenvolvedores fazem checkout de arquivos
  • 🔴 Sem internet = sem commits
  • 🔴 Servidor cai = toda equipe parada
  • 🔴 Ponto único de falha — se o HD do server morrer…
  • 🟡 Branches são lentas e caras de criar
🌐

DVCS — Distribuído

Git · Mercurial

  • 💾 Cada dev tem cópia completa do repositório
  • ✅ Trabalha offline — commit, log, branch, diff
  • ✅ Servidor cai = equipe continua trabalhando
  • ✅ Cada clone é um backup completo do projeto
  • ✅ Branches são rápidas e gratuitas
  • ✅ Suporta múltiplos fluxos de colaboração

🚀 Por que o Git dominou o mercado?

Git é 10-100× mais rápido que SVN em operações locais · Branches são criadas em milissegundos · GitHub (2008) + GitLab (2011) democratizaram a colaboração open-source · O modelo de Pull Request revolucionou o code review em equipes. Hoje: 94% dos devs usam Git (Stack Overflow Survey 2023).

🌐 GCS na Prática — O Ecossistema Git

Como os conceitos de GCS se mapeiam nas ferramentas que usaremos

Conceito GCS Ferramenta / Comando Git Quando ocorre Identificação de IC git add arquivo.c Ao "marcar" arquivos para o controle de versão Registro de mudança (commit) git commit -m "mensagem" A cada unidade lógica de trabalho concluída Baseline git tag v1.0.0 Ao lançar uma versão estável Ramificação git branch feature/login Ao iniciar nova funcionalidade ou correção Auditoria / Log de Mudanças git log --graph --oneline Para revisar histórico completo do projeto Integração / Release git merge + GitHub Actions Ao unir código e acionar o pipeline de CI/CD

🔄 Fluxo Básico GCS com Git

  • 1. Criar/clonar repositório (init/clone)
  • 2. Modificar ICs (editar arquivos)
  • 3. Adicionar ao staging (git add)
  • 4. Registrar mudança (git commit)
  • 5. Compartilhar (git push)
  • 6. Marcar versões (git tag)
