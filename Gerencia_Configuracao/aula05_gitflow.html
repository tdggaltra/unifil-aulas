🎭 O Problema que o GitFlow resolve

Imagine 6 devs num projeto. Dev A trabalha na feature de pagamento, Dev B corrige um bug crítico em produção, Dev C prepara a versão 2.0. Como garantir que um não atrapalha o outro? Que o hotfix chega em produção hoje, sem esperar a feature ficar pronta? Que o deploy sempre sai de um estado testado e aprovado? O GitFlow define exatamente isso — um protocolo de ramificação para equipes.

🎯 Objetivo desta aula

Dominar o modelo GitFlow com seus 5 tipos de branches, executar o ciclo completo com a CLI git flow e receber as especificações do Trabalho Prático — que usa GitFlow como requisito obrigatório.

🌊 O que é GitFlow?

Vincent Driessen, 2010 — o artigo que mudou o desenvolvimento em equipe

GitFlow é um conjunto de convenções sobre como nomear e usar branches — não é uma ferramenta, é um protocolo de comunicação entre devs.

📜 Origem e Contexto

  • Autor: Vincent Driessen — artigo "A Successful Git Branching Model" (2010)
  • Surgiu da necessidade de organizar ciclos de release com múltiplas equipes
  • Publicado em nvie.com — tornou-se referência mundial
  • Adotado por GitHub, GitLab e Atlassian como base de seus fluxos
  • Em 2020 o próprio Driessen adicionou nota: "para projetos com releases agendadas"
  • Melhor alternativa ao trunk-based para times maiores

✅ Quando usar GitFlow?

  • Produto com ciclos de release bem definidos (v1.0, v2.0…)
  • Times com múltiplos devs trabalhando em paralelo
  • Necessidade de suporte a versões anteriores (hotfixes)
  • Ambientes separados: dev, staging, produção
  • Projetos acadêmicos com equipes — como o nosso Trabalho!
  • Empresas com processo de QA antes de cada release

🌿 Os 5 Tipos de Branch — Papéis e Regras

Cada branch tem uma responsabilidade única e imutável

🏁 main Permanente · Protegido
Contém sempre código em produção, estável e testado. Cada commit em main é marcado com uma tag de versão (v1.0.0). Ninguém commita direto — só via merge de release/* ou hotfix/*.
🔄 develop Permanente · Base de integração
O "tronco de desenvolvimento". Features integradas aqui antes de ir para produção. Representa o estado mais recente e aprovado do próximo release. Toda feature/* nasce e morre aqui.
✨ feature/nome-da-feature Temporário · Nasce e morre em develop
Uma por funcionalidade. Criada a partir de develop, mergeada de volta em develop quando pronta. Nunca toca main diretamente. Ex: feature/autenticacao-jwt, feature/relatorio-pdf.
📦 release/1.0.0 Temporário · Nasce em develop, fecha em main + develop
Fase de preparação do próximo release. Apenas bugfixes, ajustes de documentação e versão. Nenhuma feature nova! Ao fechar, merge vai para main (com tag) e para develop (para propagar os fixes).
🚨 hotfix/descricao-urgente Temporário · Nasce em main, fecha em main + develop
Correção emergencial de bug em produção. É o único branch que nasce direto do main. Ao fechar, merge vai para main (com nova tag PATCH) e para develop. Prioridade máxima — nada espera um hotfix.

📊 Diagrama GitFlow — Visão Completa

Como os branches se relacionam ao longo do tempo

🏁 main + develop

Branches permanentes — existem para sempre no projeto. Nunca são deletados.

✨ feature, release, hotfix

Branches temporários — deletados após o merge. Cada um tem missão e duração definidas.

📦 release + 🚨 hotfix

Fecham em dois lugares: main (tag de versão) E develop (propagar mudanças). Nunca esquecer o develop!

⚙️ git-flow CLI — Automatizando o Fluxo

Instalação e comandos essenciais da extensão oficial

Instalação

Linux/Mac: apt install git-flow ou brew install git-flow
Windows: incluído no Git for Windows · git flow version para testar

git flow init

Inicializa o GitFlow no repositório. Cria o branch develop e configura os prefixos. Pressione Enter para aceitar os padrões.

git flow feature start login

Cria feature/login a partir de develop e faz checkout automaticamente.

git flow feature finish login

Mergeia feature/login em develop, deleta o branch e faz checkout em develop.

git flow feature publish login

Publica o branch feature no remote — para que colegas possam colaborar ou revisar via PR.

📦 Release e 🚨 Hotfix com git-flow

Preparando versões e respondendo a emergências em produção

📦

Release Branch

Nasce em develop · Fecha em main + develop

release workflow
# Criar release 1.1.0 $ git flow release start 1.1.0 Switched to 'release/1.1.0' from 'develop' # Apenas bugfixes e ajustes de versão $ git commit -am "chore: bump version to 1.1.0" $ git commit -am "fix: ajusta erro no relatório PDF" # Fechar: merge em main + tag + merge em develop $ git flow release finish 1.1.0 Switched to branch 'main' Merge: release/1.1.0 → main (tag: v1.1.0) Switched to branch 'develop' Merge: release/1.1.0 → develop Deleted branch release/1.1.0 # Publicar tag no GitHub $ git push origin main develop --tags
🚨

Hotfix Branch

Nasce em main · Fecha em main + develop

hotfix workflow
# Bug crítico em produção! Criar hotfix $ git flow hotfix start 1.1.1 Switched to 'hotfix/1.1.1' from 'main' # Corrigir o bug o quanto antes $ git commit -am "fix: corrige falha crítica no pagamento" # Fechar: merge em main (nova tag PATCH) + develop $ git flow hotfix finish 1.1.1 Switched to branch 'main' Merge: hotfix/1.1.1 → main (tag: v1.1.1) Switched to branch 'develop' Merge: hotfix/1.1.1 → develop Deleted branch hotfix/1.1.1 # Deploy imediato para produção! $ git push origin main develop --tags

💡 Por que o hotfix também vai para develop?

Se o fix fosse só para main, na próxima release o bug voltaria! O merge em develop garante que a correção seja carregada por todas as features e releases futuras — princípio do GitFlow de manter os dois branches permanentes sempre sincronizados.

🧪 Laboratório — GitFlow Completo

Da inicialização ao hotfix em produção

bash — roteiro completo aula 05
## PARTE 1: Inicializar GitFlow $ cd ~/lab-git && git flow init # Enter em tudo $ git branch # deve mostrar main e develop ## PARTE 2: Desenvolver uma feature $ git flow feature start calculadora-v2 $ echo "float div(float a,float b){return b?a/b:0;}" >> calc.c $ git commit -am "feat: adiciona divisão com proteção zero" $ git commit -am "test: adiciona casos de teste para divisão" $ git flow feature finish calculadora-v2 $ git log --oneline --graph # feature mergeada em develop ## PARTE 3: Preparar release $ git flow release start 1.0.0 $ git commit -am "chore: bump version 1.0.0 no README" $ git flow release finish 1.0.0 # digita mensagem da tag: "Release 1.0.0" $ git log --oneline --graph --all # veja main com tag v1.0.0! ## PARTE 4: Hotfix urgente $ git switch main # confirma que está em produção $ git flow hotfix start 1.0.1 $ git commit -am "fix: corrige overflow na função soma" $ git flow hotfix finish 1.0.1 $ git push origin main develop --tags # publica tudo no GitHub $ git log --oneline --graph --all # diagrama completo!
