O Problema: Múltiplas Identidades

Quando um desenvolvedor possui contas GitHub distintas — pessoal, empresa, docência — o Git precisa saber qual chave usar para cada repositório, sob pena de autenticar com a conta errada ou falhar com Permission denied (publickey).

Nossas 3 Contas

  • tdggalvao — conta pessoal
  • tdggaltra — GALTRA empresa
  • TiagoUnifil — docência UniFil

O que pode dar errado

  • Push para repo errado
  • Commits assinados com e-mail incorreto
  • Permission denied ao clonar
  • Confusão no histórico de contribuições
⚠️ Atenção: Se você tiver apenas uma chave SSH genérica (id_rsa), o GitHub sempre vai autenticar como a conta que tem essa chave cadastrada — independente do repositório.

Fundamentos do SSH

SSH (Secure Shell) usa criptografia assimétrica: um par de chaves matematicamente relacionadas — chave privada (fica no seu computador) e chave pública (cadastrada no GitHub). Apenas quem possui a chave privada consegue provar identidade.

Algoritmo Recomendado

Use ed25519 — mais moderno, seguro e com chaves menores que o RSA.

  • Curva elíptica (ECC)
  • Resistente a ataques modernos
  • Padrão atual do GitHub

SSH vs HTTPS

  • SSH: sem senha a cada push
  • SSH: mais seguro em CI/CD
  • HTTPS: exige token (PAT)
  • SSH: ideal para múltiplas contas

Regra de nomenclatura das chaves

  • Nome explícito: id_github_<contexto> — nunca use id_rsa genérico
  • Flag -f define o caminho e nome do arquivo gerado
  • Flag -C é apenas um comentário — use o e-mail da conta
  • Cada conta no GitHub recebe apenas a sua chave pública (.pub)
💡 Dica: Mantenha uma planilha ou README privado mapeando conta ↔ chave ↔ e-mail. Com o tempo é fácil perder o controle.

Passo 1 — Verificar Chaves Existentes

Antes de gerar novas chaves, verifique o que já existe para não sobrescrever acidentalmente.

Terminal

ls -al ~/.ssh

# Saída típica se já houver chaves:
-rw-------  1 tiago  staff  411  id_github_pessoal
-rw-r--r--  1 tiago  staff  104  id_github_pessoal.pub
-rw-------  1 tiago  staff  411  id_github_galtra
-rw-r--r--  1 tiago  staff  104  id_github_galtra.pub

Arquivo Privado

  • Sem extensão: id_github_pessoal
  • Permissão 600 (somente dono lê)
  • Nunca compartilhe ou exponha
  • Fica apenas no seu computador

Arquivo Público

  • Extensão .pub
  • Permissão 644
  • É o que você cola no GitHub
  • Pode ser compartilhado sem risco
⚠️ Se existir um id_rsa ou id_ed25519 genérico (sem nome de conta), não apague — apenas crie as novas chaves nomeadas nos próximos passos.

Passo 2 — Gerar uma Chave por Conta

Execute um comando para cada conta. A flag -f nomeia o arquivo, -t ed25519 define o algoritmo e -C é o comentário (e-mail).

Terminal — gerar as 3 chaves

# Conta pessoal — tdggalvao
ssh-keygen -t ed25519 -C "tdggalvao@gmail.com" -f ~/.ssh/id_github_pessoal

# Conta GALTRA — tdggaltra
ssh-keygen -t ed25519 -C "tdggalvao1@gmail.com" -f ~/.ssh/id_github_galtra

# Conta UniFil — TiagoUnifil
ssh-keygen -t ed25519 -C "tdggalvao2@gmail.com" -f ~/.ssh/id_github_unifil
💡 Cada comando pedirá uma passphrase (senha da chave). Em ambiente de desenvolvimento controlado pode deixar vazia pressionando Enter. Em produção/CI, configure a passphrase.

Resultado — arquivos gerados

  • ~/.ssh/id_github_pessoal  &  id_github_pessoal.pub
  • ~/.ssh/id_github_galtra  &  id_github_galtra.pub
  • ~/.ssh/id_github_unifil  &  id_github_unifil.pub

Passo 3 — Adicionar ao ssh-agent

O ssh-agent é um processo em segundo plano que armazena as chaves descriptografadas em memória, evitando que você redigite a passphrase a cada operação Git.

Terminal

# 1. Iniciar o agente (se não estiver rodando)
eval "$(ssh-agent -s)"
# Saída esperada: Agent pid 12345

# 2. Adicionar as três chaves
ssh-add ~/.ssh/id_github_pessoal
ssh-add ~/.ssh/id_github_galtra
ssh-add ~/.ssh/id_github_unifil

# 3. Confirmar que as 3 foram carregadas
ssh-add -l

macOS — persistência

  • Use --apple-use-keychain
  • Salva no Keychain do sistema
  • Persiste após reinicialização
  • Configure em ~/.ssh/config com AddKeysToAgent yes

Comandos úteis

  • ssh-add -l — listar chaves
  • ssh-add -D — remover todas
  • ssh-add -d arquivo — remover uma
  • ssh-agent -k — matar o agente

Passo 4 — Cadastrar no GitHub

Repita para cada uma das 3 contas. Copie a chave pública e adicione em Settings → SSH and GPG keys → New SSH key.

Copiar a chave pública de cada conta

# Copiar para o terminal (cole no GitHub)
cat ~/.ssh/id_github_pessoal.pub    # → logue como tdggalvao
cat ~/.ssh/id_github_galtra.pub     # → logue como tdggaltra
cat ~/.ssh/id_github_unifil.pub     # → logue como TiagoUnifil

# macOS: copiar direto para a área de transferência
pbcopy < ~/.ssh/id_github_galtra.pub

① Faça login na conta GitHub correta

github.com → seu avatar → Settings

② Acesse SSH and GPG keys

Menu lateral → SSH and GPG keys → New SSH key

↓

③ Cole a chave pública (.pub)

Dê um título descritivo, ex: "MacBook Pro Tiago 2025"

↓

④ Clique em Add SSH key

Repita para as outras 2 contas

Passo 5 — Criar o ~/.ssh/config

Este é o coração da solução. O arquivo config define aliases para o GitHub: cada alias aponta para uma chave SSH diferente. Quando o Git usa a URL git@alias-aqui:..., o SSH sabe exatamente qual chave (e portanto qual conta) usar.

~/.ssh/config
# ── Conta pessoal — tdggalvao ──────────────────
Host github-pessoal
  HostName      github.com
  User         git
  IdentityFile ~/.ssh/id_github_pessoal
  AddKeysToAgent yes
# ── Conta GALTRA — tdggaltra ────────────────────
Host github-galtra
  HostName      github.com
  User         git
  IdentityFile ~/.ssh/id_github_galtra
  AddKeysToAgent yes
# ── Conta UniFil — TiagoUnifil ──────────────────
Host github-unifil
  HostName      github.com
  User         git
  IdentityFile ~/.ssh/id_github_unifil
  AddKeysToAgent yes
💡 O campo Host é um apelido livre. O HostName é sempre github.com. A mágica é que cada alias usa um IdentityFile diferente.

💡 Na primeira conexão o terminal perguntará sobre a autenticidade do host. Digite yes para aceitar e adicionar github.com ao known_hosts.

Passo 7 — Usar o Alias nas URLs

A regra é simples: na URL do repositório, substitua github.com pelo alias definido no config. O Git vai usar o alias para rotear para a chave e conta corretas.

Clonar repositórios

# Repo pessoal
git clone git@github-pessoal:tdggalvao/meu-projeto.git

# Repo GALTRA
git clone git@github-galtra:tdggaltra/crmpro.git

# Repo UniFil
git clone git@github-unifil:TiagoUnifil/aulas.git

Corrigir remote em repositório já existente

# Ver a URL atual
git remote -v

# Trocar para o alias correto
git remote set-url origin git@github-galtra:tdggaltra/crmpro.git

# Verificar
git remote -v
💡 Padrão da URL: git@<ALIAS>:<usuario>/<repositorio>.git — o ALIAS substitui github.com, o resto permanece igual.

Passo 8 — Identidade dos Commits

A chave SSH autentica o acesso, mas o nome e e-mail do commit são metadados separados. Configure-os por repositório (sem --global) para garantir que cada commit seja assinado com a identidade correta.

Configurar identidade por repositório

# Dentro do repo da GALTRA
git config user.name  "Tiago Galvão"
git config user.email "tdggalvao1@gmail.com"

# Dentro do repo da UniFil
git config user.name  "Prof. Tiago Galvão"
git config user.email "tdggalvao2@gmail.com"

# Verificar a configuração local
git config --list --local

Config local vs global

  • --local: só para o repo atual
  • --global: padrão para todos os repos
  • Local sobrepõe o global
  • Arquivo: .git/config

⚠️ Evite --global

  • Define um e-mail padrão global
  • Pode vazar para repos da conta errada
  • Commits aparecem no perfil incorreto
  • Difícil auditar depois

Bônus — Múltiplos E-mails em uma Conta

O GitHub permite associar vários e-mails a uma única conta. Isso serve para que commits feitos com e-mails diferentes ainda apareçam vinculados ao seu perfil.

E-mail primário

tdggalvao@gmail.com — recebe notificações, é exibido no perfil público

+

E-mails secundários

tiago@galtra.com.br, tiago@codingtoyou.com — apenas associam commits ao perfil

↓

Resultado

Commits de qualquer e-mail vinculado aparecem no seu histórico de contribuições

Como adicionar

  • GitHub → Settings → Emails
  • Add email address
  • Confirme pelo link enviado
  • Marque como verificado

O que muda

  • Commits do e-mail alternativo aparecem no perfil
  • A chave SSH ainda define qual conta faz o push
  • O e-mail do commit é só metadado
  • Útil para projetos open source pessoais

Conclusão

Principais Pontos:

01

Cada conta precisa de uma chave SSH nomeada — nunca use id_rsa genérico para múltiplas contas

02

O ~/.ssh/config é o coração da solução — define aliases que roteiam para a chave correta

03

Substitua github.com pelo alias nas URLs — isso é tudo que muda no uso diário

04

SSH autentica o acesso · git config user.email assina o commit — são camadas independentes
