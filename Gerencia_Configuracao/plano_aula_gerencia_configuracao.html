🌿 Unidade II — Branches e GitFlow

Aulas 04 e 05 · 20/05 · 27/05/2026

🔀

Ramificações e Modelo GitFlow

Trabalhar em paralelo sem bagunçar o código de produção

🌲 Aula 04 — 20/05 · Branches e Merge

  • → Por que usar branches? Isolamento de features
  • → branch · checkout · switch
  • → Merge fast-forward vs 3-way merge
  • → Conflitos: identificar e resolver
  • → rebase — histórico linear
  • → Estratégias: merge commit, squash

⚙️ Aula 05 — 27/05 · GitFlow

  • → O modelo GitFlow (Vincent Driessen, 2010)
  • → Branch main — produção estável
  • → Branch develop — integração
  • → feature/* — novas funcionalidades
  • → release/* — preparação de versão
  • → hotfix/* — correção emergencial
  • → Instalação e uso do git-flow CLI

🌊 Fluxo GitFlow — Visão Geral

main ←← hotfix/* develop ←← feature/* & ←→ release/*
git flow init  →  git flow feature start minha-feature  →  git flow feature finish minha-feature  →  git flow release start 1.0.0

🔁 Unidade III — Colaboração e Pull Requests

Aulas 06 e 07 · 03/06 · 10/06/2026

👥

Trabalho em Equipe e Revisão de Código

Fork, contribuição, Pull Requests e Code Review profissional

🍴 Aula 06 — 03/06 · Colaboração

  • Fork — cópia pessoal de repositório alheio
  • Fluxo: Fork → Clone → Branch → Push → PR
  • Rebase interativo — git rebase -i
  • Squash — unir commits antes do PR
  • Stash — guardar trabalho inacabado
  • Tags — marcar versões estáveis

📬 Aula 07 — 10/06 · Pull Requests

  • Anatomia de um Pull Request de qualidade
  • Criação: título, descrição, reviewers, labels
  • Code Review: o que verificar e como dar feedback
  • Branch protection rules e required reviewers
  • Templates de PR com .github/PULL_REQUEST_TEMPLATE.md
  • Draft PRs e WIP (Work in Progress)

📋 Boas Práticas de PR

  • → Um propósito por PR
  • → Máx. 400 linhas alteradas
  • → Título no imperativo
  • → Screenshots para UI

🔍 Code Review: O Que Verificar

  • → Lógica e correção
  • → Legibilidade do código
  • → Testes cobrem o change?
  • → Segurança e performance

✅ Critérios de Aprovação

  • → CI/CD passando (verde)
  • → Ao menos 1 reviewer
  • → Sem conflitos
  • → Branch atualizada

💬 Etiqueta de Code Review — Regra de Ouro

Critique o código, nunca a pessoa. Use perguntas: "O que você acha de…?" em vez de afirmações: "Isso está errado." Elogie soluções elegantes. O objetivo do review é melhorar o produto, não mostrar quem é mais esperto.

  • → Secrets e variáveis de ambiente
  • → Environments com approvals
  • → Monitoramento pós-deploy

📄 Exemplo — Workflow CI Básico

# .github/workflows/ci.yml
name: CI Pipeline
on: [push, pull_request]
jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm install && npm test
