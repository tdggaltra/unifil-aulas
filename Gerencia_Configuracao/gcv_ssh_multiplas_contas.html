⚠️ O Problema — Múltiplas Identidades

Por que uma única chave SSH genérica causa conflitos

🎯 Situação comum no mundo real

Um desenvolvedor possui três contas GitHub distintas: uma pessoal, uma da empresa e uma da instituição de ensino. Com uma única chave SSH genérica, o GitHub sempre autentica como a mesma conta — independente do repositório.

👤 Conta Pessoal

  • Projetos próprios e open source
  • Portfólio e experimentos
  • user: dev-pessoal
  • email: dev.pessoal@gmail.com

🏢 Conta Empresa

  • Repositórios corporativos
  • Projetos de clientes
  • user: dev-empresa
  • email: dev@techcorp.com.br

🎓 Conta Faculdade

  • Trabalhos e aulas
  • Projetos acadêmicos
  • user: DevFaculdade
  • email: dev@universidade.edu.br

❌ O que acontece sem a configuração correta

Erro típico: Permission denied (publickey) ao tentar fazer push em outro perfil.
Problema silencioso: push funciona, mas vai para a conta errada — commits aparecem no perfil incorreto.
Raiz do problema: sem o arquivo ~/.ssh/config, o SSH usa a primeira chave disponível, não a correta.

🔐 Fundamentos do SSH

Criptografia assimétrica — como o GitHub autentica sem senha

SSH usa um par de chaves matematicamente relacionadas: a chave privada fica no seu computador, a pública fica no GitHub. Quem possui a privada prova a identidade sem transmitir segredos.

🔑 Algoritmo ed25519 (recomendado)

  • Criptografia de curva elíptica (ECC)
  • Mais seguro e rápido que RSA
  • Chaves menores — 68 caracteres
  • Padrão atual do GitHub
  • Flag: -t ed25519

⚡ SSH vs HTTPS

  • SSH: sem senha a cada push
  • SSH: ideal para múltiplas contas
  • SSH: melhor para pipelines CI/CD
  • HTTPS: exige token PAT por conta
  • HTTPS: difícil alternar contas

📁 Par de arquivos gerado por conta

Chave privada (sem extensão) → ~/.ssh/id_github_pessoal — permissão 600 — NUNCA compartilhe!
Chave pública (.pub) → ~/.ssh/id_github_pessoal.pub — permissão 644 — essa vai para o GitHub.

🛠️ Passos 1 e 2 — Verificar e Gerar Chaves

Uma chave nomeada para cada conta — jamais use id_rsa genérico

🔍 Passo 1 — Verificar chaves existentes

Antes de gerar, confira o que já existe para não sobrescrever acidentalmente. Se houver um id_rsa ou id_ed25519 genérico, não apague — apenas crie as novas com nomes específicos.

Terminal
# Listar arquivos na pasta SSH
ls -al ~/.ssh

# Saída esperada após configuração:
-rw-------  id_github_pessoal       # ← privada (600)
-rw-r--r--  id_github_pessoal.pub   # ← pública  (644)
-rw-------  id_github_empresa
-rw-r--r--  id_github_empresa.pub
-rw-------  id_github_faculdade
-rw-r--r--  id_github_faculdade.pub
-rw-r--r--  config                  # ← arquivo de alias
-rw-r--r--  known_hosts

⚙️ Passo 2 — Gerar uma chave por conta (flag -f nomeia o arquivo)

  • ssh-keygen -t ed25519 -C "dev.pessoal@gmail.com" -f ~/.ssh/id_github_pessoal
  • ssh-keygen -t ed25519 -C "dev@techcorp.com.br" -f ~/.ssh/id_github_empresa
  • ssh-keygen -t ed25519 -C "dev@universidade.edu.br" -f ~/.ssh/id_github_faculdade

📌 Significado das flags

  • -t ed25519 → algoritmo (curva elíptica)
  • -C "email" → comentário identificador
  • -f ~/.ssh/nome → nome do arquivo gerado
  • Enter duas vezes → sem passphrase (desenvolvimento)

⚠️ Por que nomear com -f?

  • Sem -f, gera id_ed25519 genérico
  • Sobrescreve chave anterior!
  • Nome explícito: fácil saber qual conta
  • Ex: id_github_empresa → inequívoco

⚡ Passo 3 — Adicionar ao ssh-agent

O agente gerencia as chaves em memória — sem redigitar passphrase

O ssh-agent é um processo em segundo plano que armazena as chaves descriptografadas em memória. Com ele, você não precisa redigitar a passphrase a cada operação Git.
# 1. Iniciar o agente eval "$(ssh-agent -s)" # Saída: Agent pid 12345 # 2. Adicionar as 3 chaves (--apple-use-keychain persiste no Keychain) ssh-add --apple-use-keychain ~/.ssh/id_github_pessoal ssh-add --apple-use-keychain ~/.ssh/id_github_empresa ssh-add --apple-use-keychain ~/.ssh/id_github_faculdade # 3. Confirmar que as 3 foram carregadas ssh-add -l # Saída: 256 SHA256:... dev.pessoal@gmail.com (ED25519) # 256 SHA256:... dev@techcorp.com.br (ED25519) # 256 SHA256:... dev@universidade.edu.br (ED25519)
# 1. Iniciar o agente eval "$(ssh-agent -s)" # 2. Adicionar as 3 chaves ssh-add ~/.ssh/id_github_pessoal ssh-add ~/.ssh/id_github_empresa ssh-add ~/.ssh/id_github_faculdade # 3. Para persistência (adicione ao ~/.bashrc ou ~/.zshrc): if [ -z "$SSH_AUTH_SOCK" ]; then eval "$(ssh-agent -s)" ssh-add ~/.ssh/id_github_pessoal ssh-add ~/.ssh/id_github_empresa ssh-add ~/.ssh/id_github_faculdade fi # Verificar ssh-add -l

🛠️ Comandos úteis do agente

  • ssh-add -l → listar chaves carregadas
  • ssh-add -D → remover todas as chaves
  • ssh-add -d ~/.ssh/arquivo → remover uma
  • ssh-agent -k → encerrar o agente

✅ Como saber que funcionou

  • ssh-add -l lista os 3 fingerprints
  • Cada linha mostra o e-mail do comentário
  • Se aparecer "no identities" → passo 2 falhou
  • macOS: Keychain guarda entre reinicializações

☁️ Passo 4 — Cadastrar no GitHub

Cada conta recebe apenas a sua chave pública .pub

📋 Regra fundamental

Repita o procedimento para cada uma das 3 contas. Entre no GitHub com a conta correta antes de adicionar a chave. Uma chave pública cadastrada na conta errada não autentica a conta certa!

Terminal — copiar a chave pública
# Exibir no terminal (copie manualmente)
cat ~/.ssh/id_github_pessoal.pub
cat ~/.ssh/id_github_empresa.pub
cat ~/.ssh/id_github_faculdade.pub

# macOS: copiar direto para a área de transferência
pbcopy < ~/.ssh/id_github_pessoal.pub   # logue como dev-pessoal
pbcopy < ~/.ssh/id_github_empresa.pub   # logue como dev-empresa
pbcopy < ~/.ssh/id_github_faculdade.pub # logue como DevFaculdade

① Faça login na conta GitHub correta

github.com → clique no avatar → Settings

② Acesse a seção de chaves SSH

Menu lateral → SSH and GPG keys → botão New SSH key

③ Cole a chave pública (.pub) e nomeie

Título sugestivo: "Notebook Pessoal 2026" — facilita identificar de qual máquina é

④ Confirme com Add SSH key

O GitHub pode pedir senha de confirmação. Repita os passos ①–④ para as outras 2 contas.

⚙️ Passo 5 — O arquivo ~/.ssh/config

O coração da solução — aliases que roteiam para a chave certa

O arquivo ~/.ssh/config define aliases para o GitHub. Cada alias aponta para um IdentityFile diferente. O Git usa o alias na URL para saber qual chave (e portanto qual conta) usar.
~/.ssh/config
# ── Conta pessoal — dev-pessoal ─────────────────────────
Host github-pessoal
  HostName      github.com
  User          git
  IdentityFile  ~/.ssh/id_github_pessoal
  AddKeysToAgent yes

# ── Conta empresa — dev-empresa ─────────────────────────
Host github-empresa
  HostName      github.com
  User          git
  IdentityFile  ~/.ssh/id_github_empresa
  AddKeysToAgent yes

# ── Conta faculdade — DevFaculdade ──────────────────────
Host github-faculdade
  HostName      github.com
  User          git
  IdentityFile  ~/.ssh/id_github_faculdade
  AddKeysToAgent yes

🔍 Anatomia do bloco Host

  • Host → apelido livre (você define o nome)
  • HostName → sempre github.com
  • User → sempre git
  • IdentityFile → chave privada desta conta
  • AddKeysToAgent → carrega no agente automaticamente

⚠️ Criar o arquivo

  • Se não existir: touch ~/.ssh/config
  • Editar: nano ~/.ssh/config
  • Permissão correta: chmod 600 ~/.ssh/config
  • Verificar: cat ~/.ssh/config

🧪 Passos 6 e 7 — Testar e Usar os Aliases

Verificar cada conexão e aplicar os aliases nas URLs dos repositórios

✅ Passo 6 — Testar as 3 conexões

  • Cada teste responde com o usuário correto da conta correspondente
  • ssh -T git@github-pessoal → deve retornar: Hi dev-pessoal! You've successfully authenticated...
  • ssh -T git@github-empresa → deve retornar: Hi dev-empresa! You've successfully authenticated...
  • ssh -T git@github-faculdade → deve retornar: Hi DevFaculdade! You've successfully authenticated...

Passo 7 — Usar alias nas URLs dos repositórios
# Regra: substitua github.com pelo alias nas URLs
# Padrão: git@<ALIAS>:<usuario-github>/<repositorio>.git

# Clonar com a conta correta
git clone git@github-pessoal:dev-pessoal/meu-projeto.git
git clone git@github-empresa:dev-empresa/sistema-crm.git
git clone git@github-faculdade:DevFaculdade/trabalho-final.git

# Corrigir remote em repositório já existente
git remote -v                                        # ver atual
git remote set-url origin git@github-empresa:dev-empresa/sistema-crm.git
git remote -v                                        # confirmar

✍️ Passo 8 — Identidade dos Commits

SSH autentica o acesso · git config assina o commit — camadas independentes

📐 Conceito fundamental

A chave SSH autentica quem pode fazer o push. O git config user.email define com qual identidade o commit é assinado. Configure por repositório (sem --global) para garantir que cada commit use o e-mail correto.

Terminal — configurar identidade por repositório
# Dentro do repositório da EMPRESA
git config user.name  "Nome Sobrenome"
git config user.email "dev@techcorp.com.br"

# Dentro do repositório da FACULDADE
git config user.name  "Nome Sobrenome (Acadêmico)"
git config user.email "dev@universidade.edu.br"

# Verificar a configuração local do repositório atual
git config --list --local

# Verificar quem vai assinar o próximo commit
git config user.name
git config user.email

🔍 Local vs Global

  • --local: só para o repo atual (arquivo .git/config)
  • --global: padrão para todos os repos (~/.gitconfig)
  • Local sempre sobrepõe o global
  • Não configure --global para múltiplas contas!

⚠️ Evite --global com múltiplas contas

  • Define e-mail padrão para todos os repos
  • Commits da empresa podem ir com e-mail pessoal
  • Commits aparecem no perfil incorreto no GitHub
  • Difícil de auditar e corrigir depois

4. 💬 Análise de Cenário:

Um desenvolvedor fez git push com sucesso, mas o commit apareceu no perfil errado no GitHub (conta pessoal em vez da empresa). Qual é a causa mais provável e como corrigir?

💡 Resposta:
Causa: A chave SSH autenticou corretamente (push funcionou), mas o git config user.email estava configurado com o e-mail pessoal (configuração global).
Correção: Dentro do repositório da empresa, executar git config user.email "dev@techcorp.com.br" (sem --global). Para corrigir commits já feitos, usar git commit --amend --reset-author.
