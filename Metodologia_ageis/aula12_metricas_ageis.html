📊 Por Que Métricas Ágeis?

O que não se mede, não se melhora

Métricas servem para APRENDER e MELHORAR, não para PUNIR! 🎯

✅ Boas Razões para Métricas

  • Entender capacidade do time
  • Prever entregas futuras
  • Identificar gargalos
  • Melhorar processos
  • Tomar decisões baseadas em dados
  • Celebrar melhorias

❌ Más Razões

  • ❌ Comparar times
  • ❌ Avaliar desempenho individual
  • ❌ Punir time "lento"
  • ❌ Microgerenciar
  • ❌ Gaming (manipular números)
  • ❌ Pressionar por velocidade

🎯 Princípios das Métricas Ágeis

  • 1. Transparência: Métricas visíveis para todos
  • 2. Contexto: Números sem contexto não significam nada
  • 3. Tendências: Importa mais a direção que o valor absoluto
  • 4. Múltiplas Métricas: Uma métrica sozinha engana
  • 5. Ação: Métricas devem levar a melhorias concretas

⚠️ Goodhart's Law

"Quando uma medida se torna uma meta, ela deixa de ser uma boa medida."

Exemplo: Se você medir time por "Story Points entregues" e transformar isso em meta,
time vai inflar estimativas para "bater meta". Métrica se corrompe! 📉

"Métricas são como faróis: iluminam o caminho, mas não dirigem o barco!"

🚀 Velocity (Velocidade)

Quanto o time entrega por Sprint?

🚀 O Que É Velocity?

Quantidade de Story Points (ou outras unidades) que o time consegue completar em um Sprint.
Medida de capacidade do time, não de produtividade individual!
Usada para prever quanto trabalho cabe em futuros Sprints.

42 pts

Velocity Média do Time (últimos 3 Sprints)

📊 Exemplo: Calculando Velocity

Sprint 1: Planejado 50 pts → Completado 38 pts ✅
Sprint 2: Planejado 45 pts → Completado 42 pts ✅
Sprint 3: Planejado 48 pts → Completado 46 pts ✅

Velocity Média: (38 + 42 + 46) / 3 = 42 pontos

Sprint 4 Planning: Time deve planejar ~42 pontos
(Pode variar entre 38-46 pts dependendo do contexto)

✅ Bom Uso de Velocity

  • ✓ Previsibilidade de entregas
  • ✓ Planejamento realista
  • ✓ Identificar instabilidade
  • ✓ Comunicar com stakeholders
  • ✓ Melhorar estimativas

❌ Mau Uso

  • ❌ Comparar times diferentes
  • ❌ Pressão para aumentar velocity
  • ❌ KPI de performance
  • ❌ Bônus baseado em velocity
  • ❌ Ignorar contexto (férias, etc)

⚡ Velocity NÃO É Constante!

Velocity varia por muitos motivos:
• Férias, feriados, doenças
• Novo membro no time (velocity cai temporariamente)
• Débito técnico resolvido (velocity sobe depois)
• Complexidade das histórias varia

Use MÉDIA dos últimos 3-5 Sprints, não apenas o último!

"Velocity é FERRAMENTA do time, não ARMA da gerência!"

📉 Burndown Chart

Quanto trabalho resta no Sprint?

📉 O Que É Burndown Chart?

Gráfico que mostra trabalho restante ao longo do tempo.
Eixo Y: Story Points/Horas restantes
Eixo X: Dias do Sprint
Linha ideal: Reta descendente do total até zero.

📊 Sprint Burndown Chart (Exemplo Visual)

0pts
Dias do Sprint →
Linha Ideal
Trabalho Real

✅ Padrões Saudáveis

  • Declínio Suave: Trabalho diminui gradualmente
  • Próximo da Ideal: Real acompanha linha ideal
  • Chega a Zero: Todo trabalho completo no fim

⚠️ Padrões de Alerta

  • Flat Line: Trabalho não diminui (impedimentos?)
  • Muito Acima Ideal: Overcommitment
  • Scope Creep: Linha sobe (trabalho adicionado)
  • Queda Abrupta Final: Work in Progress demais

🎯 Como Usar Burndown no Daily

1. SM atualiza burndown diariamente (após Daily)
2. Time visualiza progresso em direção a Sprint Goal
3. Se fora da ideal: Time discute ajustes
4. Não é para cobrar! É para inspecionar e adaptar

Burndown é ferramenta do TIME para auto-organização!

"Burndown mostra o presente. Use-o para ajustar o curso, não para culpar!"

📈 Burnup Chart

Quanto trabalho foi completado?

📈 O Que É Burnup Chart?

Gráfico que mostra trabalho completado ao longo do tempo.
2 linhas:
• Linha de Total Scope (pode mudar!)
• Linha de Work Completed (sempre sobe)

Quando as duas linhas se encontram = Sprint completo!

Burnup mostra PROGRESSO positivo! Mais motivador que Burndown! 📈

✅ Vantagens do Burnup

  • Mostra scope creep claramente
  • Celebra trabalho completado
  • Mais motivador (linha sobe!)
  • Útil para releases longos
  • Visualiza mudanças de escopo

📊 Quando Usar Burnup?

  • Projetos com escopo variável
  • Releases multi-Sprint
  • Stakeholders querem ver progresso
  • Time desmotivado (burndown negativo)
  • Product Goal de longo prazo

📊 Burndown vs Burnup: Mesma Situação

Situação: Sprint de 50 pts, 10 dias

Dia 5:
• Burndown: Ainda restam 25 pts (metade!) 😐
• Burnup: Já completamos 25 pts! 🎉

Mesmo dado, percepção diferente!
Burnup enfatiza conquista. Burndown enfatiza trabalho restante.

Use o que motiva SEU time! Ou use AMBOS! 📊

🔄 Burnup para Release Planning

Ideal para prever quando um Product Goal será atingido:

• Linha superior = Total do Product Backlog para o Goal
• Linha inferior = Story Points completados (acumula a cada Sprint)
• Extrapole a linha de progresso → estima quantos Sprints faltam

Se escopo mudar (linha superior sobe/desce), impacto é imediatamente visível!

"Burndown pergunta: Quanto falta? Burnup celebra: Quanto fizemos!"

⏱️ Lead Time e Cycle Time

Quanto tempo leva para entregar valor?

⏱️ O Que São Lead Time e Cycle Time?

Métricas de fluxo que medem tempo de entrega:

Lead Time: Tempo desde REQUEST até DELIVERY
(Cliente pede → Cliente recebe)

Cycle Time: Tempo desde START até DONE
(Time começa → Time completa)

📊 Exemplo: Lead Time vs Cycle Time

User Story: "Login com Google"

• 01/03: PO adiciona ao Product Backlog (REQUEST)
• 05/03: Time começa a trabalhar (START)
• 12/03: Feature completa e em produção (DONE)

Lead Time: 01/03 → 12/03 = 11 dias
Cycle Time: 05/03 → 12/03 = 7 dias

Diferença: Lead Time inclui tempo esperando no backlog!

🎯 Por Que Medir Lead Time?

  • Perspectiva do CLIENTE
  • Quanto tempo espera por valor?
  • Inclui todo o processo
  • Útil para SLA com cliente
  • Indica eficiência total

🎯 Por Que Medir Cycle Time?

  • Perspectiva do TIME
  • Quanto tempo leva para fazer?
  • Exclui espera no backlog
  • Identifica gargalos internos
  • Melhora processo do time

📊 Como Reduzir Lead/Cycle Time?

  • 1. Quebrar User Stories: Histórias menores = ciclos mais curtos
  • 2. Limitar WIP: Menos trabalho simultâneo = entrega mais rápida
  • 3. Automatizar: CI/CD reduz tempo de deploy
  • 4. Reduzir Handoffs: Menos transferências entre pessoas
  • 5. Priorizar Backlog: Itens importantes não esperam semanas
"Lead Time longo? Cliente espera demais. Cycle Time longo? Time tem gargalo!"
