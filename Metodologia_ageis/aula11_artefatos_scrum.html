📦 Os 3 Artefatos do Scrum

Transparência e visibilidade do trabalho

3 Artefatos = Papéis + Eventos + Artefatos = Scrum Completo! 🎯

📖 O Que São Artefatos?

Artefatos do Scrum representam trabalho ou valor.
São projetados para maximizar transparência de informações-chave.
Assim, todos têm o mesmo entendimento do artefato.

📋 Product Backlog

  • O QUÊ: Lista ordenada de tudo que é necessário no produto
  • QUEM: PO é dono
  • OBJETIVO: Única fonte da verdade

📝 Sprint Backlog

  • O QUÊ: Plano para entregar Sprint Goal
  • QUEM: Developers são donos
  • OBJETIVO: Visibilidade do Sprint

✅ Incremento

  • O QUÊ: Produto funcionando
  • QUEM: Time cria
  • OBJETIVO: Entregar valor

🎯 Cada Artefato Tem Um "Commitment"

  • 📋 Product Backlog → Commitment: Product Goal
  • 📝 Sprint Backlog → Commitment: Sprint Goal
  • ✅ Incremento → Commitment: Definition of Done

Esses commitments reforçam empirismo e valores Scrum!

Artefatos sem transparência = trabalho invisível = desperdício!

📋 Product Backlog

A fonte única da verdade

📋 O Que É Product Backlog?

Lista ordenada e emergente de tudo que é necessário para melhorar o produto.
É a única fonte de trabalho realizado pelo Scrum Team.
Nunca está completo. Evolui constantemente.

✅ Características

  • Ordenado: Por valor, risco, dependência
  • Emergente: Nunca completo, sempre evoluindo
  • Transparente: Todos podem ver
  • Dinâmico: Muda a cada Sprint
  • Único: Um backlog por produto

📦 O Que Contém?

  • User Stories (funcionalidades)
  • Bugs e correções
  • Melhorias técnicas
  • Pesquisas/Spikes
  • Débito técnico

📝 Estrutura de Item do Product Backlog

Título: Login com Google
Descrição: Como usuário, quero fazer login com minha conta Google para não precisar criar nova senha
Critérios de Aceitação:
• Botão "Login com Google" visível na tela de login
• Redireciona para OAuth do Google
• Cria conta automaticamente se primeiro login
• Redireciona para dashboard após sucesso

Valor de Negócio: Alto (reduz fricção no onboarding)
Estimativa: 8 pontos
Prioridade: #5 no backlog

👑 PO é Dono do Product Backlog

Responsabilidades do PO:
• Ordenar itens por valor
• Refinar itens (adicionar detalhes)
• Garantir visibilidade e transparência
• Aceitar ou rejeitar sugestões de mudança

PO pode delegar refinamento, mas permanece responsável!

Product Backlog nunca está "pronto". Se está parado, produto está morrendo!

📝 Sprint Backlog

O plano do Sprint

📝 O Que É Sprint Backlog?

Composto de: Sprint Goal + Itens selecionados + Plano para entregá-los
É um plano feito POR Developers PARA Developers.
Atualizado em tempo real durante o Sprint.

Sprint Backlog = Sprint Goal + O QUÊ fazer + COMO fazer 🎯

🎯 Sprint Goal

  • Objetivo único do Sprint
  • Criado no Planning
  • Direciona Developers
  • Permite flexibilidade
  • Exemplo: "Usuários podem fazer checkout completo"

📋 Itens Selecionados

  • Do Product Backlog
  • Selecionados no Planning
  • Suficientes para Sprint Goal
  • Podem ser ajustados
  • Negociados com PO

📊 Exemplo de Sprint Backlog (Visual)

Sprint Goal: "Usuários podem comprar produtos com cartão de crédito"

📋 Itens do Product Backlog:
✓ User Story: Adicionar produtos ao carrinho (8 pts)
✓ User Story: Integração com gateway pagamento (13 pts)
✓ User Story: Página de confirmação pedido (5 pts)
✓ Bug: Corrigir cálculo de frete (3 pts)

🛠️ Plano (Tarefas):
• Backend: Criar API carrinho (João, 6h)
• Frontend: Tela carrinho (Maria, 8h)
• Integração: Stripe SDK (Pedro, 10h)
• QA: Testes end-to-end (Ana, 4h)
• DevOps: Deploy staging (Carlos, 2h)

Total Estimado: 29 pontos | 30 horas trabalho

👥 Developers São Donos do Sprint Backlog

Durante o Sprint, APENAS Developers podem:
• Adicionar/remover tarefas
• Atualizar progresso
• Reestimar trabalho restante
• Reorganizar plano

PO NÃO pode mudar Sprint Backlog sem consenso dos Developers!

Sprint Backlog é transparente, atualizado e visível a todos. Qualquer um pode VER, só Developers podem MUDAR!

✅ Incremento

O produto funcionando

✅ O Que É Incremento?

Um degrau concreto em direção ao Product Goal.
Cada Incremento é aditivo a todos os Incrementos anteriores.
Deve estar "Done" (atender Definition of Done).
Múltiplos Incrementos podem ser criados em um Sprint.

Incremento = Soma de TUDO que está "Done" até agora! 📦

✅ Características

  • Funcionando: Produto real, não protótipo
  • Usável: Pode ser usado por usuário
  • Testado: Passou em todos os testes
  • "Done": Atende DoD
  • Potencialmente Liberável: Pode ir para produção

🚀 Quando Criar?

  • Pelo menos 1 por Sprint
  • Pode ter vários por Sprint
  • Deve ser apresentado no Review
  • PO decide quando liberar
  • Incrementos se somam!

📈 Exemplo: Incrementos Acumulativos

Sprint 1 - Incremento:
• Login funcionando
• Cadastro de usuário
• Página inicial
= Produto tem 3 features

Sprint 2 - Incremento:
• Catálogo de produtos
• Busca
= Produto agora tem 5 features (3 + 2)

Sprint 3 - Incremento:
• Carrinho de compras
• Checkout
= Produto agora tem 7 features (5 + 2)

💡 Cada Incremento ADICIONA ao anterior! Produto cresce continuamente.

🎯 "Potencialmente Liberável" ≠ "Deve Ser Liberado"

Potencialmente Liberável: Incremento PODE ir para produção (está pronto)
Decisão de Liberar: PO decide QUANDO liberar

Time pode criar vários Incrementos no Sprint, mas PO decide se/quando deployar!
Algumas organizações deployam a cada Sprint. Outras acumulam para releases maiores.

Incremento que não está "Done" NÃO pode ser liberado. Incremento "Done" CAN (mas não MUST) ser liberado!

✅ Definition of Done (DoD)

Quando algo está realmente "pronto"?

✅ O Que É Definition of Done?

Descrição formal do estado do Incremento quando ele atende as medidas de qualidade requeridas para o produto.
Checklist que TODO item do backlog deve atender para ser considerado "Done".
Cria transparência do que significa "completo".

DoD = Contrato de qualidade! Se não passou em tudo, NÃO está Done! ✅

📋 Exemplo: Definition of Done (E-commerce)

  • Código escrito e revisado (Code Review aprovado)
  • Testes unitários criados e passando (>80% cobertura)
  • Testes de integração passando
  • Testes de aceitação (E2E) passando
  • Documentação técnica atualizada
  • Sem bugs críticos ou bloqueadores
  • Deploy em ambiente de staging bem-sucedido
  • Aceito pelo Product Owner
  • Atende critérios de acessibilidade (WCAG 2.1)
  • Performance dentro dos SLAs (< 2s carregamento)
  • Segurança verificada (OWASP top 10)
  • Pode ser demonstrado ao usuário final

Se 1 item não for atendido, trabalho NÃO está "Done"!

✅ Bom DoD

  • ✓ Específico e verificável
  • ✓ Mesmo padrão para tudo
  • ✓ Criado pelo Scrum Team
  • ✓ Evolui ao longo do tempo
  • ✓ Público e visível
  • ✓ 8-15 itens (não muito longo)

❌ DoD Ruim

  • ❌ Vago: "Código de qualidade"
  • ❌ Diferente por tipo de trabalho
  • ❌ Imposto por gerência
  • ❌ Nunca atualizado
  • ❌ Escondido/não compartilhado
  • ❌ Lista gigante (50+ itens)

⚙️ DoD Evolui Com o Time

Time Novo: DoD pode ser simples (código funciona, testado manualmente)
Time Maduro: DoD inclui CI/CD, testes automatizados, segurança, performance

Na Retrospectiva: Time discute se DoD precisa evoluir!
Adicionar critério aumenta qualidade. Remover critério = técnico débito!

"Done" significa DONE! Não "quase pronto", "95% completo". É binário: Done ou Not Done!
