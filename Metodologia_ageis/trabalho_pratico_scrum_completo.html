🎯 1. Introdução e Objetivos

Este trabalho tem como objetivo aplicar todo o framework Scrum aprendido em aula em um projeto real desenvolvido em grupo. Vocês irão simular pelo menos 1 Sprint completa, aplicando todos os papéis, eventos e artefatos do Scrum.

🎓 Objetivos de Aprendizagem

  • Aplicar os 3 papéis do Scrum (Product Owner, Scrum Master, Developers)
  • Executar os 5 eventos do Scrum (Sprint, Planning, Daily, Review, Retrospective)
  • Criar e manter os 3 artefatos (Product Backlog, Sprint Backlog, Incremento)
  • Coletar e analisar métricas ágeis (Velocity, Burndown, Lead Time)
  • Trabalhar em equipe auto-organizada
  • Entregar incremento funcional de valor

⚠️ Importante

Este trabalho é prático e hands-on! Não é apenas documentação teórica. Vocês precisam realmente executar uma Sprint, desenvolver algo funcional, fazer reuniões reais (mesmo que virtuais), e entregar um incremento que demonstre valor.

👥 2. Formação da Equipe

2.1 Composição

  • Tamanho: 4 a 5 pessoas por equipe
  • Definição de papéis: Cada equipe deve ter 1 PO, 1 SM, e 2-3 Developers
  • Rotação: Opcional - vocês podem rotacionar papéis em Sprints futuras

2.2 Responsabilidades por Papel

Papel Responsável no Trabalho
Product Owner (PO) • Criar e priorizar Product Backlog
• Definir Sprint Goal
• Aceitar ou recusar incremento
• Representar stakeholders
Scrum Master (SM) • Facilitar todos os eventos
• Remover impedimentos
• Atualizar métricas (Burndown)
• Garantir aderência ao Scrum
Developers • Estimar User Stories
• Desenvolver incremento
• Auto-organizar trabalho
• Garantir qualidade (testes, code review)

  • Nome completo de cada membro

💡 3. Escolha do Projeto

3.1 Critérios para o Projeto

✅ O projeto deve:

  • Ser factível de desenvolver em 2 semanas (1 Sprint)
  • Ter valor claro (resolver problema real ou hipotético)
  • Permitir incremento funcional (algo demonstrável)
  • Ser divisível em User Stories pequenas
  • Engajar todos os membros da equipe

3.2 Exemplos de Tipos de Projetos

Escolha UM dos tipos abaixo (ou sugira outro com aprovação do professor):

📱 App Mobile

Aplicativo mobile simples (React Native, Flutter, ou web app responsivo)

Ex: App de lista de tarefas, controle financeiro pessoal, agenda de estudos

🌐 Sistema Web

Aplicação web (HTML/CSS/JS + backend simples ou Firebase)

Ex: Sistema de reserva de salas, catálogo de produtos, blog com CMS

🎮 Game/Gamificação

Jogo simples ou sistema gamificado

Ex: Quiz interativo, jogo da memória, sistema de pontos/badges

🤖 Chatbot/Automação

Bot ou sistema de automação

Ex: Chatbot FAQ, bot de notificações, scraper de dados

📊 Dashboard/Analytics

Painel de visualização de dados

Ex: Dashboard COVID, análise de vendas, métricas pessoais

🛒 E-commerce

Loja virtual simples

Ex: Catálogo de produtos, carrinho, checkout simulado

📚 Sistema Educacional

Plataforma de ensino/aprendizado

Ex: Quiz de estudos, flashcards, sistema de notas

🏥 Saúde/Bem-estar

App de saúde ou fitness

Ex: Rastreador de água, diário alimentar, treinos

📋 4. Product Backlog Inicial

4.1 Criação do Backlog

O Product Owner deve criar um Product Backlog inicial com 15 a 20 User Stories priorizadas.

📝 Formato de User Story

Como [papel/usuário],
Eu quero [funcionalidade],
Para [benefício/valor]

Exemplo:
Como estudante,
Eu quero criar flashcards de estudo,
Para memorizar conteúdos para provas

4.2 Critérios INVEST

Cada User Story deve seguir INVEST:

Letra Critério O que significa
I Independent Independente de outras histórias
N Negotiable Detalhes podem ser discutidos
V Valuable Entrega valor ao usuário
E Estimable Pode ser estimada pelo time
S Small Pequena o suficiente para caber em Sprint
T Testable Tem critérios de aceitação claros

4.3 Estrutura de Cada Item do Backlog

Para cada User Story, incluir:

  • ID: Número único (US-001, US-002, etc)
  • Título: Nome descritivo
  • User Story: Formato "Como... Eu quero... Para..."
  • Prioridade: Alta, Média ou Baixa (ordenar por valor!)
  • Estimativa: Story Points (Fibonacci: 1, 2, 3, 5, 8, 13)
  • Critérios de Aceitação: Checklist de quando está "Done"

📦 Entrega 3: Product Backlog Inicial

Documento: Planilha ou documento com 15-20 User Stories

  • Todas as stories no formato correto
  • Ordenadas por prioridade (mais valiosas no topo)
  • Com estimativas em Story Points
  • Critérios de aceitação para as top 5
  • Definition of Ready aplicada

Data de entrega: 25/03/2026

🎯 5. Sprint Planning

5.1 Definições da Sprint

  • Duração da Sprint: 2 semanas (10 dias úteis)
  • Período: 25/03 a 07/04 (exemplo - ajustar conforme necessário)
  • Capacidade: Baseada em disponibilidade do time

5.2 Sprint Planning - O QUÊ?

  • PO apresenta top items do Product Backlog
  • Time discute e esclarece dúvidas
  • Time define Sprint Goal (meta única e coerente)
  • Time seleciona User Stories para a Sprint
  • Usar Velocity (se houver histórico) ou capacidade estimada

    • 5.3 Sprint Planning - COMO?

  • Quebrar cada User Story em tarefas técnicas
  • Estimar tarefas em horas (opcional)
  • Identificar dependências técnicas
  • Criar Sprint Backlog
  • Todos comprometidos com Sprint Goal

    • 📦 Entrega 4: Sprint Planning

    Documentos:

    • Sprint Goal: Frase clara do que será alcançado
    • Sprint Backlog: User Stories selecionadas + tarefas
    • Estimativa inicial: Total de Story Points planejados
    • Evidência da reunião: Foto/print do planning (pode ser virtual)
    • Quadro Kanban: Setup inicial (To Do, Doing, Done + WIP limits)

    Data de entrega: 25/03/2026 (após planning)

    ⚡ 6. Execução da Sprint e Daily Scrum

    6.1 Daily Scrum

    Realizar Daily Scrum todos os dias (ou pelo menos 3x por semana se time for remoto).

    ⏱️ Formato do Daily

    • Duração: Máximo 15 minutos
    • Horário: Mesmo horário todos os dias
    • 3 Perguntas:
      • O que fiz ontem?
      • O que farei hoje?
      • Há algum impedimento?
    • Scrum Master: Atualiza Burndown após Daily

    6.2 Gestão do Sprint Backlog

  • Atualizar status das tarefas diariamente
  • Mover cards no quadro Kanban
  • Registrar impedimentos
  • SM trabalha para remover bloqueios
  • Time colabora e se auto-organiza

    • 6.3 Desenvolvimento do Incremento

    Durante a Sprint, o time deve:

    • Desenvolver código funcional
    • Fazer testes (unitários, integração)
    • Code review entre membros
    • Integração contínua (commits frequentes)
    • Documentar decisões técnicas importantes

    📦 Entrega 5: Evidências de Execução

    Documentos e evidências:

    • Registro de Dailys: Ata resumida de pelo menos 5 Dailys
      • Data, participantes
      • Resumo do que cada um falou
      • Impedimentos identificados
    • Burndown Chart: Atualizado diariamente (gráfico)
    • Quadro Kanban: Prints do quadro em 3 momentos (início, meio, fim da Sprint)
    • Impediments Log: Lista de impedimentos e como foram resolvidos

    Data de entrega: 07/04/2026 (fim da Sprint)

    🎉 7. Sprint Review

    7.1 Preparação

    ⚠️ Importante

    Sprint Review NÃO é PowerPoint! É demonstração ao vivo do incremento funcional.

    Preparem o ambiente com antecedência. Testem tudo antes!

    7.2 Estrutura da Review (15 minutos por equipe)

    Etapa Tempo Atividade
    Abertura 2 min PO recapitula Sprint Goal e o que foi planejado
    Demo 8 min Developers demonstram incremento FUNCIONANDO
    Discussão 4 min Feedback, perguntas, próximos passos
    Encerramento 1 min Agradecimentos e próxima Sprint (se houver)

    7.3 O Que Demonstrar

  • Apenas User Stories "Done" (completas!)
  • Software funcionando (não slides!)
  • Valor de negócio entregue
  • Funcionalidades do ponto de vista do usuário
  • Se algo não ficou pronto, explicar por quê (sem drama!)

    • 📦 Entrega 6: Sprint Review

    Apresentação em aula + documentos:

    • Demo ao vivo: 15 minutos de apresentação
    • Incremento funcional: Software rodando (link, repo, ou executável)
    • Resumo da Sprint:
      • Sprint Goal
      • User Stories planejadas vs completadas
      • Velocity alcançado
      • Principais desafios
    • Feedback recebido: Notas do que stakeholders (colegas/professor) comentaram

    Data de entrega: 07/04/2026 (apresentação em aula)

    🔄 8. Sprint Retrospective

    8.1 Objetivo

    Inspecionar como foi a Sprint em termos de pessoas, processos e ferramentas, e definir melhorias concretas.

    🎯 Prime Directive

    "Independentemente do que descobrirmos, entendemos e realmente acreditamos que todos fizeram o melhor trabalho possível, dado o que sabiam na época, suas habilidades e capacidades, os recursos disponíveis e a situação em questão."

    8.2 Técnica Sugerida: Start-Stop-Continue

    Categoria Pergunta
    Start (Começar) O que devemos COMEÇAR a fazer?
    Stop (Parar) O que devemos PARAR de fazer?
    Continue (Continuar) O que devemos CONTINUAR fazendo?

    8.3 Estrutura da Retro (1 hora)

  • Set the Stage (5 min): Check-in, criar ambiente seguro
  • Gather Data (15 min): O que aconteceu? Linha do tempo, fatos
  • Generate Insights (20 min): Por que aconteceu? Causas raiz
  • Decide What to Do (15 min): Ações concretas (máximo 3!)
  • Close (5 min): Apreciações, próximos passos

    • 📦 Entrega 7: Sprint Retrospective

    Documento: Relatório da Retrospectiva

    • Participantes: Quem estava presente
    • Start-Stop-Continue: Lista de itens em cada categoria
    • Ações de Melhoria: 1 a 3 ações concretas
      • O que fazer?
      • Quem é responsável?
      • Quando implementar?
    • Aprendizados: Principais insights do time

    Data de entrega: 08/04/2026

    📊 9. Métricas Ágeis

    9.1 Métricas Obrigatórias

    Cada equipe deve coletar e analisar as seguintes métricas:

    Métrica Como Calcular Para Quê
    Velocity Total de Story Points completados na Sprint Capacidade do time, planejamento futuro
    Burndown Chart Gráfico de Story Points restantes por dia Acompanhar progresso diário
    Lead Time Tempo desde criação até Done de cada story Tempo de entrega ao cliente
    Cycle Time Tempo desde "In Progress" até Done Eficiência de execução

    9.2 Análise das Métricas

    Para cada métrica, responder:

    • Qual foi o valor obtido?
    • O que isso indica sobre o processo do time?
    • Houve alguma anomalia ou surpresa?
    • Como podemos melhorar esse número?

    📦 Entrega 8: Relatório de Métricas

    Documento: Análise de métricas da Sprint

    • Velocity: Valor e interpretação
    • Burndown Chart: Gráfico com análise (estava no caminho? Por quê?)
    • Lead Time médio: Cálculo e análise
    • Cycle Time médio: Cálculo e análise
    • Conclusões: O que as métricas revelam sobre o processo?
    • Ações: Como usar esses dados para melhorar?

    Data de entrega: 08/04/2026

    ✅ 10. Critérios de Avaliação

    Critério Peso Descrição
    Incremento Funcional 25% Qualidade do software entregue, funcionalidades working, valor demonstrado
    Aplicação do Scrum 25% Execução correta dos papéis, eventos e artefatos. Evidências de práticas
    Product Backlog 10% Qualidade das User Stories (INVEST), priorização, critérios de aceitação
    Métricas e Melhoria 15% Coleta de métricas, análise, uso de dados para decisões, Retrospective
    Apresentação (Review) 15% Clareza, organização, demonstração efetiva, gestão do tempo
    Documentação 10% Completude, organização, evidências (prints, atas, registros)

    🌟 Critérios de Excelência (Nota 10)

    • Incremento funcional de alta qualidade, impressionante
    • Scrum aplicado com maestria (todos eventos executados corretamente)
    • Métricas bem coletadas e usadas para decisões reais
    • Melhoria contínua demonstrada (Retrospective com ações concretas)
    • Apresentação profissional e engajadora
    • Documentação impecável e bem organizada
    • Colaboração exemplar entre membros

    💡 12. Dicas para Sucesso

    🎯 Comece Simples

    Escolha projeto simples mas interessante. Melhor entregar MVP funcional que projeto ambicioso incompleto!

    📋 User Stories Pequenas

    Quebrem histórias grandes! Idealmente cada story deve ser completável em 1-2 dias.

    🤝 Colaboração Real

    Façam reuniões reais (presenciais ou virtuais). Documentem com prints/fotos!

    📊 Meçam Desde Início

    Não deixem métricas para última hora. Coletem dados diariamente!

    🔄 Daily de Verdade

    15 minutos, todo dia, mesmo horário. Se remoto, use WhatsApp/Discord mas seja consistente!

    ✅ Definition of Done

    Definam DoD clara desde início. Código funcional + testado + documentado mínimo.

    🎭 Demo Impressionante

    Ensaiem a Sprint Review! Testem tudo antes. Demo ao vivo, não PowerPoint!

    📸 Evidências

    Tirem prints do quadro Kanban, reuniões, commits. Evidências contam pontos!
