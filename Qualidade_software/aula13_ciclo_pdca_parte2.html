🌍 PDCA em Diferentes Contextos

Aplicabilidade universal

Um Ciclo, Mil Aplicações! 🌐

💻 Software Development

Problema: Build time muito lento (15 minutos)
PLAN: Analisar dependências, identificar gargalos → Hipótese: cache ineficiente
DO: Implementar layer caching Docker + npm cache (1 projeto piloto)
CHECK: Build time: 15min → 3min 🎉
ACT: Padronizar para todos 40 microserviços, documentar, treinar time

🏭 Manufatura (Toyota)

Problema: Taxa defeito 5% em linha montagem
PLAN: Diagrama Ishikawa → Causa: calibração sensor incorreta
DO: Recalibrar sensores, implementar poka-yoke (à prova de erros)
CHECK: Taxa defeito: 5% → 0.8% em 1 mês
ACT: Calibração semanal obrigatória, treinamento operadores, auditoria mensal

🏥 Saúde (Hospital)

Problema: Tempo espera triagem emergência 3h (meta <1h)
PLAN: VSM processo triagem → Gargalo: registro manual dados
DO: Implementar sistema digital triagem (piloto 1 turno)
CHECK: Tempo espera: 3h → 45min, satisfação +35%
ACT: Escalar para 3 turnos, treinar 100% enfermeiros, monitoramento contínuo

🎓 Educação (Universidade)

Problema: Taxa reprovação Cálculo I = 40% (meta <20%)
PLAN: Análise notas → Dificuldade: derivadas, falta base
DO: Aulas de reforço pré-requisitos + monitoria 2x/semana (1 turma piloto)
CHECK: Reprovação turma piloto: 40% → 15% 🎉
ACT: Reforço obrigatório para TODAS turmas, contratar 3 monitores, avaliação diagnóstica

"O PDCA funciona porque reflete como humanos aprendem: tentativa → erro → análise → ajuste → sucesso!" 🧠

⚖️ PDCA vs DMAIC

Comparação com Six Sigma

Dois Gigantes da Melhoria! 🏆

🔄 PDCA

  • Origem: 1920s (Shewhart)
  • Filosofia: Lean, Kaizen
  • Passos: 4 (simples)
  • Duração típica: Dias a semanas
  • Foco: Melhoria rápida incremental
  • Complexidade: Baixa
  • Ferramentas: 7 básicas qualidade

📊 DMAIC

  • Origem: 1980s (Motorola)
  • Filosofia: Six Sigma
  • Passos: 5 (Define-Measure-Analyze-Improve-Control)
  • Duração típica: Semanas a meses
  • Foco: Redução variação, qualidade estatística
  • Complexidade: Alta
  • Ferramentas: Estatística avançada
Aspecto PDCA DMAIC
Melhor para Problemas conhecidos, melhorias rápidas Problemas complexos, alta variação
Treinamento Horas (qualquer pessoa) Semanas (Green/Black Belts)
Investimento Baixo (tempo, ferramentas simples) Alto (consultoria, software estatístico)
Velocidade Rápida (ciclos curtos) Lenta (análise profunda)
Precisão Boa (pragmática) Excelente (estatística rigorosa)
Cultura Todos participam Especialistas lideram

🎯 Quando usar cada um?

✅ Use PDCA quando:
  • Problema bem definido
  • Precisa resultado rápido
  • Time não tem treinamento Six Sigma
  • Orçamento limitado
  • Quer cultura participativa
  • Melhorias incrementais
✅ Use DMAIC quando:
  • Problema complexo, causa desconhecida
  • Alta variação no processo
  • Tem Green/Black Belts
  • Investimento disponível
  • Meta: 3.4 defeitos/milhão (Six Sigma)
  • Breakthrough necessário

🤝 Melhor estratégia: Usar AMBOS!

• PDCA: 80% dos problemas (melhoria contínua diária)
• DMAIC: 20% dos problemas (projetos críticos)
• Não são concorrentes — são complementares! 🤝

🏃 PDCA Ágil (Scrum)

Integrando com metodologias ágeis

Scrum = PDCA em Ciclos de 2 Semanas! 🔄
Scrum Event Mapeamento PDCA Objetivo
Sprint Planning PLAN Definir o que fazer e como
Sprint Execution DO Executar trabalho planejado
Sprint Review CHECK (Produto) Validar incremento com stakeholders
Sprint Retrospective CHECK (Processo) Analisar como time trabalhou
Action Items / Next Sprint ACT Implementar melhorias identificadas

💼 Caso Real: Time Scrum com Velocity Instável

Problema:
• Velocity oscila 25-45 SP/sprint (instável!)
• Dificulta planejamento releases
• Time frustrado, stakeholders inseguros

PLAN (Sprint Planning + Retro):
• Analisar últimos 6 sprints
• Identificar causa: Interrupções não planejadas (47/sprint!)
• Hipótese: Falta buffer para suporte produção
• Plano: Reservar 20% capacity para interrupções

DO (Sprint N):
• Sprint Planning: Comprometer 32 SP (não 40)
• Reservar 8 SP para interrupções
• Rastrear interrupções em Jira

CHECK (Review + Retro):
• Velocity: 36 SP entregues (planejado 32 + 4 extras)
• Interrupções: 12 (dentro do buffer!)
• Time menos estressado, entregaram comprometido

ACT (Próximos Sprints):
• Padronizar: SEMPRE reservar 20% buffer
• Após 4 sprints: Velocity estável 36-39 SP 🎉
• Novas melhorias: reduzir interrupções (novo PDCA!)

🎯 PDCA em TODOS os níveis Scrum

• Sprint (2 semanas): PDCA do trabalho
• Release (3 meses): PDCA do produto
• Anual: PDCA da estratégia
Melhoria contínua em múltiplas escalas! ♾️

⚠️ Armadilhas Comuns do PDCA

E como evitá-las

Aprenda com Erros dos Outros! 🚫
❌ Armadilha 1: Pular o PLAN

Erro: "Já sei o problema, vamos logo fazer!"
Consequência: Resolver sintoma, não causa. Problema volta.
Solução: SEMPRE investir no Plan. 50% do tempo total!

❌ Armadilha 2: Escalar sem Piloto

Erro: Implementar mudança em TODA empresa de uma vez
Consequência: Se falhar, impacto massivo. Resistência.
Solução: Sempre fazer piloto pequeno primeiro (DO)!

❌ Armadilha 3: Não Coletar Dados

Erro: Executar mudança mas não medir resultados
Consequência: Como saber se funcionou? Achismo!
Solução: Definir métricas no Plan, coletar no Do, analisar no Check

❌ Armadilha 4: Não Padronizar Sucesso

Erro: Conseguir melhoria mas não documentar/treinar
Consequência: Melhoria some quando pessoa-chave sai
Solução: ACT = Padronizar! Documentar, treinar, auditar.

❌ Armadilha 5: Parar no Primeiro Ciclo

Erro: Resolver 1 problema e parar
Consequência: Não é melhoria CONTÍNUA!
Solução: ACT → novo PLAN. Sempre tem algo para melhorar!

❌ Armadilha 6: Atacar Sintomas, Não Causas

Erro: "Bug em produção? Vamos fazer code review!" (mas bugs vêm de falta testes)
Consequência: Problema persiste, frustração aumenta
Solução: Usar 5 Porquês, Ishikawa para achar CAUSA RAIZ

"A definição de insanidade é fazer a mesma coisa repetidamente e esperar resultados diferentes!" — Albert Einstein 🧠

✅ Boas Práticas PDCA

Receita para o sucesso

Faça Certo desde o Início! 🎯
✅ Prática 1: Investir no PLAN (50% do tempo)

• Use 5 Porquês, Ishikawa, Pareto
• Valide causa raiz com DADOS
• Defina metas SMART
• Plano de ação 5W2H completo

✅ Prática 2: Começar Pequeno (Piloto)

• 1 time, 1 projeto, 1 processo
• Ambiente controlado
• Errar barato (não em produção!)
• Aprender antes de escalar

✅ Prática 3: Dados Sempre (Tudo é Métrica)

• Baseline ANTES da mudança
• Coletar DURANTE execução
• Comparar DEPOIS
• Dashboard automatizado melhor

✅ Prática 4: Documentar TUDO

• Problema, análise, plano, resultados
• Usar template A3 (1 página visual)
• Wiki/Confluence acessível
• Lições aprendidas (sucesso E falha)

✅ Prática 5: Envolver o Time

• Pessoas que fazem o trabalho sabem mais
• Ownership = comprometimento
• Workshops colaborativos
• Celebrar pequenas vitórias

✅ Prática 6: ACT Rigoroso (Follow-up)

• Padronizar ≠ Esquecer
• Auditorias periódicas (1, 3, 6 meses)
• Melhoria sustentada? Ou regrediu?
• Ajustar se necessário

"Excelência não é um ato, mas um hábito!" — Aristóteles 🌟

📝 Trabalho 3 - Projeto PDCA

Aplicação prática individual/dupla

🎯 Trabalho 3: Ciclo PDCA Completo

Valor: 10% da nota final
Formato: Individual ou dupla
Entrega: 30/03/2026 (23:59)
Extensão: 8-12 páginas + anexos

📋 O que fazer?

Escolher um problema real (trabalho, estudos, vida pessoal) e aplicar o ciclo PDCA completo para resolvê-lo.

Exemplos de problemas válidos:
  • Software: Build lento, bugs em produção, deploy manual
  • Estudos: Baixo rendimento matéria X, procrastinação
  • Trabalho: Reuniões improdutivas, documentação ruim
  • Pessoal: Tempo exercício físico, organização financeira
  • Equipe: Comunicação falha, turnover alto
Seção Conteúdo Pontos
1. Problema Descrição clara, impacto, por que resolver 1.0
2. PLAN Análise atual, causa raiz, plano 5W2H, metas SMART 3.0
3. DO Execução piloto, coleta dados, desvios 2.0
4. CHECK Análise resultados, comparação meta vs real 2.0
5. ACT Decisão (padronizar/ajustar), próximos passos 1.5
6. Conclusão Lições aprendidas, reflexão pessoal 0.5

⚠️ Importante

  • ✓ Problema REAL (não hipotético)
  • ✓ DADOS reais (métricas antes/depois)
  • ✓ Ferramentas qualidade (Ishikawa, Pareto, etc)
  • ✓ Evidências (prints, fotos, gráficos)
  • ✓ Honestidade (pode falhar! Documente aprendizado)

📄 Template A3

PDCA em 1 página visual

A3 = PDCA Visual em 1 Folha! 📄

🎯 O que é o A3?

Formato de relatório visual de 1 página (tamanho A3 = 29.7×42cm) usado pela Toyota para comunicar problema, análise, solução e resultados. Força síntese e clareza!

Estrutura Template A3 (8 seções)

1. Título/Contexto

• Nome problema
• Data, responsável
• Área/processo afetado

2. Situação Atual

• Como funciona hoje?
• Fluxograma/foto
• Métricas baseline

3. Meta/Objetivo

• Estado desejado
• Metas SMART
• Prazo

4. Análise Causa Raiz

• Diagrama Ishikawa
• 5 Porquês
• Dados comprovando

5. Contramedidas

• Plano de ação 5W2H
• Cronograma
• Responsáveis

6. Resultados

• Antes vs Depois
• Gráficos comparativos
• Atingiu meta?

7. Padronização

• Documentação
• Treinamento
• Auditoria

8. Próximos Passos

• Follow-up quando?
• Novas melhorias
• Próximo PDCA

💡 Dica de Ouro do A3

80% visual, 20% texto!
Gráficos, fotos, diagramas > parágrafos longos
Qualquer pessoa deve entender em 5 minutos! 🎨

✅ Checklists PDCA

Garantindo qualidade em cada passo

Nunca Esqueça Nada! ✅

📋 Checklist PLAN

Problema claramente definido?
Impacto quantificado?
Processo atual mapeado?
Dados baseline coletados?
Causa raiz identificada com 5 Porquês?
Diagrama Ishikawa feito?
Causa validada com dados?
Meta SMART definida?
Plano de ação 5W2H completo?

📋 Checklist DO

Piloto pequeno primeiro?
Time treinado na mudança?
Ambiente controlado?
Executando conforme plano?
Dados sendo coletados?
Desvios documentados?
Comunicação stakeholders?

📋 Checklist CHECK

Dados completos coletados?
Comparado baseline vs atual?
Meta atingida?
Desvios analisados?
Efeitos colaterais identificados?
Feedback time coletado?
Decisão tomada (continuar/ajustar/descartar)?

📋 Checklist ACT (Sucesso)

Mudança documentada?
Time todo treinado?
Processo atualizado?
Escalado para toda organização?
Auditorias agendadas? (1, 3, 6 meses)
Sucesso comunicado e celebrado?

📋 Checklist ACT (Falha)

Causa falha identificada?
Lições documentadas?
Nova hipótese formulada?
Voltar ao PLAN agendado?
