📊 7 Ferramentas Clássicas - Parte 1

Fluxograma, Ishikawa e Pareto

Ferramentas Testadas por Décadas! 📈
📐

1. Fluxograma (Flowchart)

Para quê: Mapear processos visualmente, identificar gargalos e redundâncias
Quando usar: Fase PLAN do PDCA, documentação de processos
Símbolos principais: Oval (início/fim), Retângulo (ação), Losango (decisão), Setas (fluxo)

Exemplo: Processo de Deploy

[Início] → [Executar testes] → ⟨Passou?⟩ → Sim → [Build Docker] → [Deploy staging] → ⟨Validar?⟩ → Sim → [Deploy produção] → [Fim]
⟨Passou?⟩ → Não → [Notificar dev] → [Fim]
⟨Validar?⟩ → Não → [Rollback] → [Notificar] → [Fim]

🐟

2. Diagrama Ishikawa (Espinha de Peixe / Causa e Efeito)

Para quê: Identificar causas raiz de problemas
Quando usar: Fase PLAN do PDCA, análise de incidentes
6M's típicos: Máquina, Método, Mão-de-obra, Material, Medida, Meio Ambiente

Exemplo: API lenta (>2s response)

Método: Query SQL sem índice, N+1 queries
Máquina: Servidor subdimensionado (2GB RAM)
Mão-de-obra: Dev júnior, sem code review
Medida: Não monitorando APM, sem alertas
Material: Banco dados sem otimização
Meio: Tráfego alto horário pico

Causa raiz principal: Query sem índice! 🎯

📊

3. Gráfico de Pareto (Regra 80/20)

Para quê: Priorizar problemas (poucos vitais vs muitos triviais)
Quando usar: Decisão sobre onde focar esforços
Princípio: 80% dos problemas vêm de 20% das causas

Exemplo: Bugs produção último mês (100 total)
CausaQtd%Acum
Validação frontend4545%45%
Erro integração API3030%75% ← Foco aqui!
Performance1212%87%
UI inconsistente88%95%
Outros55%100%

→ Resolver validação + API = eliminar 75% dos bugs! 🎯

📊 7 Ferramentas Clássicas - Parte 2

Histograma, Dispersão, Controle e Folha Verificação

📊

4. Histograma

Para quê: Visualizar distribuição de dados
Exemplo: Tempo resposta API

0-100ms: ████████ (40%)
100-200ms: ██████ (30%)
200-500ms: ████ (20%)
500-1s: ██ (10%)
Conclusão: 70% < 200ms ✅
📈

5. Diagrama Dispersão

Para quê: Correlação entre variáveis
Exemplo: Complexidade código × Bugs

Complexidade ciclomática vs Bugs:
• 1-10: 2 bugs/módulo
• 11-20: 8 bugs/módulo
• 21+: 25 bugs/módulo
Conclusão: Correlação forte! ⚠️
📉

6. Gráfico de Controle (Control Chart)

Para quê: Monitorar estabilidade do processo ao longo do tempo
Quando usar: Produção contínua, métricas de performance
Componentes: Linha central (média), Limites superior/inferior (±3σ)

Exemplo: Build time últimas 30 builds 
Limite Superior (LSC): 8min
─────────────────────────────────────
Média: 5min          ●  ●    ●
                  ●     ●  ●    ● ●
               ●                    ●
─────────────────────────────────────
Limite Inferior (LIC): 2min

Análise:
✅ Processo sob controle (todos dentro dos limites)
✅ Tendência estável (sem padrões anormais)
☑️

7. Folha de Verificação (Checklist)

Para quê: Coletar dados sistematicamente
Quando usar: Inspeções, testes, coleta dados repetitiva

Exemplo: Checklist Code Review
☑ Testes unitários cobrem >80% código?
☑ Sem comentários TODO/FIXME?
☑ Variáveis nomeadas claramente?
☑ Complexidade ciclomática <10?
☑ Sem código duplicado?
☑ Documentação atualizada?
☑ SonarQube Quality Gate passou?

7/7 ✅ → Aprovar PR
<7 ❌ → Request changes

🔍 Ferramentas de Análise

5 Porquês, 5W2H e FMEA

Análise Profunda de Problemas! 🔬
❓

5 Porquês (5 Whys)

Para quê: Chegar à causa raiz perguntando "Por quê?" 5 vezes
Origem: Toyota Production System (Taiichi Ohno)

Exemplo: Site fora do ar

1. Por quê site caiu? → Servidor parou de responder
2. Por quê servidor parou? → Ficou sem memória (OOM)
3. Por quê ficou sem memória? → Memory leak na aplicação
4. Por quê tem memory leak? → Conexões BD não fechadas
5. Por quê conexões não foram fechadas? → Código sem try-finally

CAUSA RAIZ: Falta de try-finally no código!
SOLUÇÃO: Adicionar connection pooling + code review

📋

5W2H (What, Why, Who, When, Where, How, How much)

Para quê: Plano de ação estruturado
Quando usar: Fase PLAN do PDCA

PerguntaExemplo: Implementar CI/CD
What (O quê?)Pipeline CI/CD com GitHub Actions
Why (Por quê?)Reduzir tempo deploy de 4h para <1h
Who (Quem?)João (DevOps) + Maria (Dev Lead)
When (Quando?)Sprint 15-16 (4 semanas)
Where (Onde?)Repositório GitHub main branch
How (Como?)1. Setup Actions 2. Config workflow 3. Teste piloto 4. Rollout
How much (Quanto?)R$0 (GitHub free) + 80h trabalho time
⚠️

FMEA (Failure Mode and Effects Analysis)

Para quê: Análise preventiva de falhas potenciais
RPN = Severidade × Ocorrência × Detecção (cada 1-10)
Quando usar: Projetos críticos, alta complexidade

Modo FalhaSODRPNAção
BD principal falhar 1023 60 Setup replica automática
API externa timeout 654 120 Circuit breaker + fallback
Disco servidor cheio 832 48 Alerta 80% + auto-cleanup logs

Prioridade: RPN >100 = URGENTE, 50-100 = ALTO, <50 = MÉDIO

Garantindo qualidade do código

Testes Automatizados = Confiança! ✅
☕

JUnit (Java)

Tipo: Unit tests
Uso: Testar métodos/classes
Cobertura típica: 80%+

🃏

Jest (JavaScript)

Tipo: Unit + Integration
Uso: React, Node.js
Feature: Snapshot testing

🌲

Cypress

Tipo: E2E (End-to-End)
Uso: Testar fluxos usuário
Feature: Time-travel debug

🤖

Selenium

Tipo: E2E browser automation
Uso: Multi-browser testing
Linguagens: Java, Python, JS

📮

Postman

Tipo: API testing
Uso: Testar endpoints REST
Feature: Collections + automation

🎭

Playwright

Tipo: E2E moderno
Uso: Cross-browser, mobile
Feature: Auto-wait, paralelo

🎯 Pirâmide de Testes Ideal

                 ╱───────╲
                ╱   E2E   ╲     10% - Cypress/Playwright
               ╱───────────╲    (Fluxos críticos completos)
              ╱             ╲
             ╱  Integration  ╲  20% - Jest/Postman
            ╱─────────────────╲ (Integrações entre módulos)
           ╱                   ╲
          ╱       Unit          ╲ 70% - JUnit/Jest
         ╱─────────────────────────╲ (Funções individuais)

Por quê essa proporção?
• Unit tests: rápidos (ms), baratos, específicos
• Integration: médios (seg), necessários, cobrem APIs
• E2E: lentos (min), caros, mas testam experiência real

💡 Regra de Ouro

Cobertura >80% ≠ Qualidade!
Prefira testes ÚTEIS a testes que só aumentam %
1 teste E2E crítico > 100 unit tests inúteis 🎯

🔍 Ferramentas de Análise de Código

Qualidade e consistência automáticas

Código Limpo = Manutenção Fácil! 🧹
🔷

SonarQube

O que faz: Análise estática de código (bugs, vulnerabilidades, code smells)
Linguagens: 25+ (Java, JS, Python, C#, etc)
Métricas: Duplicação, complexidade, cobertura testes, segurança

Quality Gate - Exemplo
MétricaCondiçãoValorStatus
Cobertura testes>80%85%✅ Passou
Duplicação código<3%1.8%✅ Passou
Bugs=02❌ Falhou
Vulnerabilidades=00✅ Passou
Code Smells<1015❌ Falhou

Quality Gate: FAILED ❌ → Bloquear merge!

🔧

ESLint (JavaScript/TypeScript)

O que faz: Linter - detecta problemas padrão código
Configurável: Regras personalizadas
Integra: VSCode, CI/CD

Exemplo regras:
• no-unused-vars
• no-console (produção)
• eqeqeq (=== em vez de ==)
• max-lines-per-function: 50
💅

Prettier (Formatação)

O que faz: Auto-formatação código
Opinionated: Poucas opções, consistência máxima
Linguagens: JS, TS, CSS, HTML, JSON

Benefício:
Zero debate sobre estilo!
Salvar arquivo → formatado ✨
Time todo = código igual
📊

CodeClimate / CodeCov

CodeClimate: Análise qualidade + maintainability score (A-F)
CodeCov: Visualização cobertura testes, badges CI/CD

Exemplo Maintainability Score

Grade A: user-service (complexidade baixa, bem testado)
Grade B: api-gateway (bom, alguns code smells)
Grade D: legacy-module (⚠️ refatorar urgente!)

→ Revisar módulos C/D em cada sprint!

🎯 Estratégia Análise Código

1. Pre-commit: ESLint + Prettier (local)
2. PR: SonarQube Quality Gate (CI/CD)
3. Semanal: CodeClimate report dashboard
= Qualidade garantida em múltiplas camadas! 🛡️

⚙️ Ferramentas de CI/CD

Integração e Deploy Contínuos

Automatizar = Acelerar + Confiabilidade! 🚀
🔵

GitHub Actions

O que faz: CI/CD nativo GitHub, workflows em YAML
Vantagens: Gratuito (public repos), integrado, marketplace actions
Uso típico: Lint → Test → Build → Deploy

name: CI/CD Pipeline
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm install
      - run: npm test
      - run: npm run build
  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - run: ./deploy.sh production
🔴

Jenkins

O que faz: CI/CD server open-source
Vantagens: Flexível, 1500+ plugins
Desvantagem: Requer manutenção server

Uso típico:
Pipelines complexos
On-premise deployment
Integração legacy systems
🦊

GitLab CI

O que faz: CI/CD integrado GitLab
Vantagens: All-in-one (Git + CI/CD + K8s)
Runners: Shared ou self-hosted

.gitlab-ci.yml:
stages: [test, deploy]
Auto DevOps disponível
Registry Docker integrado

📊 Ferramentas de Monitoramento

Observabilidade em produção

Não Pode Melhorar o Que Não Mede! 📈
📊

Grafana + Prometheus

Prometheus: Coleta métricas time-series (pull model)
Grafana: Visualização dashboards lindos
Alertas: Integra Slack, PagerDuty, email

Métricas Típicas Monitoradas
Infraestrutura:
• CPU, RAM, Disk usage
• Network I/O
• Container health
Aplicação:
• Request rate (req/s)
• Response time (p50, p95, p99)
• Error rate (%)
• Active users
🔍

Datadog

O que faz: APM + Logs + Infra monitoring
Vantagens: All-in-one, fácil setup
Desvantagem: Caro ($$$)

Features:
• Distributed tracing
• Anomaly detection (ML)
• Synthetic monitoring
🚀

New Relic

O que faz: APM completo, transaction tracing
Vantagens: Deep insights, alertas smart
Uso: Debug performance issues

Visualiza:
• Slow queries SQL
• Bottlenecks API
• Error stack traces
📝

ELK Stack (Elasticsearch + Logstash + Kibana)

Elasticsearch: Search & analytics engine
Logstash: Pipeline processamento logs
Kibana: Visualização e dashboards

Caso de Uso: Log Aggregation

Problema: 50 microserviços, logs espalhados
Solução ELK:
1. Todos serviços → logs JSON → Logstash
2. Logstash → parse, enrich → Elasticsearch
3. Kibana → search, visualize, alert

Resultado: Query "error 500 last 1h" → encontra em TODOS serviços em segundos! 🔍

🎯 Pilares da Observabilidade

1. Métricas: O QUÊ está acontecendo (Prometheus/Grafana)
2. Logs: POR QUÊ está acontecendo (ELK)
3. Traces: ONDE está o problema (Datadog/New Relic)
= Visibilidade completa! 👁️
