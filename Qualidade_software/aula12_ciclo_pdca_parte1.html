📜 Origem e História do PDCA

De Shewhart a Deming

100 Anos de Melhoria Contínua! 📚
1920s

Walter Shewhart - Bell Labs

Criou o ciclo de Shewhart: Specification → Production → Inspection → Redesign. Base do controle estatístico de qualidade (SPC).

1950

W. Edwards Deming no Japão

Deming ensina o ciclo ao Japão pós-guerra. Simplifica para Plan-Do-Check-Act. Japão adota massivamente, revoluciona indústria japonesa.

1980s

Retorno ao Ocidente

Sucesso japonês (Toyota, Sony, Honda) chama atenção. Empresas ocidentais redescobrem Deming e PDCA. Vira padrão em gestão da qualidade.

Hoje

Universal

Usado em TODO setor: manufatura, software, saúde, educação. Base de ISO 9001, Lean, Six Sigma, Kaizen. Princípio fundamental de melhoria contínua.

👨‍🔬 W. Edwards Deming (1900-1993)

Estatístico americano considerado "pai da qualidade moderna".

Contribuições:
• Ciclo PDCA
• 14 Pontos para Gestão
• Sistema de Conhecimento Profundo
• Transformou indústria japonesa

🔄 Visão Geral do Ciclo PDCA

Os 4 passos interconectados

PDCA = Método Científico Aplicado! 🔬
Passo Nome Essência Pergunta-Chave
P Plan (Planejar) Definir objetivos e métodos "O que vamos fazer e como?"
D Do (Fazer/Executar) Implementar o plano "Estamos executando conforme planejado?"
C Check (Verificar) Medir e avaliar resultados "Funcionou? Atingimos objetivos?"
A Act (Agir) Padronizar ou corrigir "Como garantir que melhoria persista?"

🎯 Princípio Fundamental

PDCA = Hipótese → Experimento → Análise → Conclusão
É o método científico aplicado à gestão! 🔬

📋 P = Plan (Planejar)

O passo mais importante!

P
PLAN - Planejar

Objetivo: Estabelecer objetivos e processos necessários para entregar resultados de acordo com requisitos do cliente e políticas da organização.

📊 5 Sub-passos do PLAN

1. Identificar o Problema

• O que está errado? "Tempo deploy 4h, meta <1h"
• Qual impacto? "20 deploys/mês × 3h extra = 60h desperdiçadas"
• Por que resolver? "Atrasar releases, frustrar equipe"
• Ferramentas: 5 Porquês, Diagrama Ishikawa, Pareto

2. Analisar o Processo Atual (Observação)

• Como funciona hoje? Mapear processo atual (fluxograma)
• Onde estão gargalos? "Testes manuais levam 2h"
• Dados: Coletar métricas (tempos, taxas erro)
• Ferramentas: VSM, Fluxograma, Cronoanálise

3. Identificar Causas Raiz

• Por que acontece? Não sintomas, CAUSAS!
• Exemplo: "Testes manuais porque não temos CI/CD"
• Validar: Causa → Efeito comprovado com dados
• Ferramentas: 5 Porquês, Ishikawa, 5W2H

4. Elaborar Plano de Ação

• O que fazer? "Implementar CI/CD com GitHub Actions"
• Como? Passos detalhados, responsáveis, prazos
• Recursos? Tempo, pessoas, budget
• Ferramentas: 5W2H (What, Why, Who, When, Where, How, How much)

5. Definir Metas e Indicadores

• Meta SMART: "Reduzir tempo deploy de 4h para <1h em 2 meses"
• KPIs: Tempo médio deploy, taxa sucesso, frequência
• Baseline: "Hoje: 4h ± 45min, 85% sucesso"
• Como medir: Dashboard automatizado (Grafana)

"Planejar bem = 50% do sucesso! Se você não sabe para onde vai, qualquer caminho serve..." 🎯

✅ D = Do (Fazer/Executar)

Hora de colocar em prática!

D
DO - Executar

Objetivo: Implementar o plano de ação, executar o processo e coletar dados para análise posterior. Fazer pequeno ANTES de escalar!

✅ O que fazer no DO

  • Executar conforme planejado: Seguir plano de ação
  • Teste piloto primeiro: Pequena escala, ambiente controlado
  • Educar e treinar: Time precisa entender mudança
  • Documentar TUDO: O que fez, quando, como
  • Coletar dados: Métricas definidas no Plan
  • Registrar desvios: O que foi diferente do planejado?

⚠️ Armadilhas comuns no DO

  • Pular Plan: "Já sei o problema, vamos logo fazer!"
  • Escalar rápido demais: Implementar em TODA empresa
  • Não treinar: Time não entende a mudança
  • Não coletar dados: Como vai avaliar no Check?
  • Desviar do plano: Improvisar sem documentar
  • Desistir cedo: "Não funcionou na 1ª semana!"

💼 Caso Prático: Implementando CI/CD

Semana 1-2: Preparação
• Treinar time em GitHub Actions (workshop 4h)
• Setup repo teste (não produção ainda!)
• Documentar workflow desejado

Semana 3-4: Piloto
• Implementar CI/CD em 1 microserviço (não crítico)
• Pipeline: lint → test → build → deploy staging
• Time 3 devs monitora de perto
• Coletar métricas: tempo deploy, taxa sucesso

Semana 5-6: Observação
• 20 deploys realizados no piloto
• Tempo médio: 45 minutos (antes 4h!) 🎉
• Taxa sucesso: 90% (antes 85%)
• 2 problemas encontrados (secrets, rollback)

Dados coletados:
• Logs completos de cada deploy
• Tempos por etapa (lint 2min, test 8min, build 15min, deploy 20min)
• Feedback time: "Muito melhor! Confiamos mais"
• Incidentes: 1 rollback necessário (processo funcionou)

🎯 Princípio-Chave do DO

Fazer pequeno primeiro!
Piloto → Aprender → Ajustar → Escalar
Nunca implemente mudanças em larga escala sem testar! 🔬

📊 C = Check (Verificar)

Analisando os resultados

C
CHECK - Verificar

Objetivo: Monitorar e medir processos e resultados contra objetivos e requisitos, relatar resultados. Comparar PLANEJADO vs REAL!

📊 O que fazer no CHECK

1. Coletar Dados

• Usar métricas definidas no Plan
• Dados quantitativos (números) + qualitativos (percepções)
• Período adequado: não julgar em 1 dia!

2. Comparar com Metas

• Meta: Tempo deploy <1h
• Real: 45 minutos ✅
• Conclusão: Meta atingida!

3. Analisar Desvios

• Positivos: O que deu melhor que esperado?
• Negativos: O que não funcionou?
• Por quê? Entender causas dos desvios

4. Identificar Efeitos Colaterais

• CI/CD melhorou deploy... mas aumentou carga servidor staging?
• Resolver problema A criou problema B?
• Benefícios inesperados? (ex: documentação melhorou)

5. Decidir: Continuar, Ajustar ou Descartar?

• Sucesso total: Prosseguir para ACT (padronizar)
• Sucesso parcial: Ajustar e repetir DO
• Falha: Voltar ao PLAN (nova análise)

Métrica Baseline Meta Real (Piloto) Status
Tempo Deploy 4h ± 45min <1h 45 min ± 8min ✅ Atingido!
Taxa Sucesso 85% >90% 90% ✅ Atingido!
Frequência Deploy 2x/semana >5x/semana 7x/semana ✅ Superado!
Satisfação Time 6/10 >8/10 9/10 ✅ Superado!

🎯 Conclusão do CHECK

✅ Todas metas atingidas ou superadas!
✅ Nenhum efeito colateral negativo significativo
✅ Time está satisfeito e confiante

Decisão: Prosseguir para ACT (padronizar e escalar)! 🚀

🎯 A = Act (Agir)

Padronizar ou corrigir

A
ACT - Agir

Objetivo: Tomar ações para melhorar continuamente o desempenho do processo. Se deu certo: PADRONIZAR! Se não: CORRIGIR e voltar ao PLAN!

✅ Cenário 1: SUCESSO

Ações no ACT:
  • Padronizar: Virar processo oficial
  • Documentar: Manual, wiki, runbooks
  • Treinar todos: Não só piloto
  • Comunicar: Compartilhar sucesso
  • Escalar: Implementar em TODA organização
  • Atualizar políticas: Novo padrão
  • Celebrar: Reconhecer time!

❌ Cenário 2: FALHA ou PARCIAL

Ações no ACT:
  • Não padronizar: Não escalar erro!
  • Analisar causa: Por que falhou?
  • Ajustar plano: Nova hipótese
  • Voltar ao PLAN: Reiniciar ciclo
  • Documentar lições: O que aprendemos?
  • Não desistir: Thomas Edison: 10.000 tentativas!

💼 Caso CI/CD: Ações no ACT

Mês 3: Padronização
• ✅ Documentar processo completo (Confluence 30 páginas)
• ✅ Criar templates GitHub Actions reutilizáveis
• ✅ Atualizar política: "Todo repo DEVE ter CI/CD"
• ✅ Workshop para TODOS devs (3 turmas, 60 pessoas)

Mês 4-6: Escalamento
• ✅ Implementar CI/CD em 15 microserviços restantes
• ✅ Time DevOps suporta migração (office hours diárias)
• ✅ Monitorar métricas: manter <1h deploy
• ✅ Ajustes finos (secrets management, notifications)

Mês 7: Nova baseline
• ✅ 100% repos com CI/CD funcionando
• ✅ Tempo deploy médio: 42 minutos (consistente!)
• ✅ Frequência: 10x/semana (era 2x)
• ✅ Taxa sucesso: 94% (era 85%)
• ✅ Satisfação time: 9.2/10

Resultado Final:
🎉 Melhoria permanente! Virou novo padrão!
🎉 Economia: 120h/mês time desenvolvimento
🎉 Novas melhorias identificadas (próximo PDCA!)

🛠️ Ferramentas da Qualidade

As 7 ferramentas clássicas

Ferramentas para Cada Passo do PDCA! 🔧
Ferramenta Para quê serve Quando usar
1. Fluxograma Mapear processo atual PLAN: Entender como funciona hoje
2. Diagrama Ishikawa (Espinha Peixe) Identificar causas raiz PLAN: Por que problema acontece?
3. Gráfico de Pareto (80/20) Priorizar causas/problemas PLAN: Onde focar esforço?
4. Histograma Distribuição de dados CHECK: Como dados se distribuem?
5. Diagrama de Dispersão Correlação entre variáveis CHECK: X influencia Y?
6. Gráfico de Controle Monitorar variação processo CHECK/ACT: Processo sob controle?
7. Folha de Verificação Coletar dados sistematicamente DO/CHECK: Registrar ocorrências

🐟 Diagrama Ishikawa (Causa e Efeito)

Problema: Deploy lento (4h)

Máquina/Tecnologia:
• Servidor staging lento
• Docker build sem cache

Método/Processo:
• Testes manuais (2h)
• Sem automação
• Deploy manual (script 50 passos)

Pessoas:
• Só 1 pessoa sabe fazer deploy
• Falta treinamento CI/CD

Medida/Dados:
• Não mede tempo por etapa
• Sem monitoramento

📊 Gráfico de Pareto (80/20)

Defeitos produção (último mês):

1. Validação frontend: 45 bugs (50%) 📊
2. Integração API: 25 bugs (28%) 📊
3. Performance BD: 12 bugs (13%)
4. UI inconsistente: 5 bugs (6%)
5. Outros: 3 bugs (3%)

Conclusão Pareto:
78% bugs vêm de 2 causas!
→ Focar em validação frontend + integração API
"Sem dados você é apenas mais uma pessoa com opinião." — W. Edwards Deming 📊

💼 Caso Completo: Bug Rate Alto

PDCA do início ao fim

🚨 Problema: Taxa de bugs produção inaceitável

Situação: E-commerce, 45 bugs/mês produção, clientes reclamando, NPS caindo

P
PLAN - 4 semanas

Identificar problema: 45 bugs/mês, meta <10 bugs/mês
Analisar: Pareto → 50% bugs são validação frontend
Causa raiz (Ishikawa): Falta testes automatizados frontend
Plano: Implementar testes E2E com Playwright
Meta SMART: Reduzir bugs validação de 23/mês para <5/mês em 3 meses

D
DO - 6 semanas

Semana 1-2: Setup Playwright, treinar 3 QAs
Semana 3-4: Escrever testes E2E para fluxo checkout (piloto)
Semana 5-6: Rodar testes em CI/CD, coletar dados
Resultado piloto: 15 testes E2E escritos, encontraram 8 bugs ANTES produção!

C
CHECK - 2 semanas

Dados (2 meses piloto):
• Bugs validação checkout: 23/mês → 4/mês ✅ (Meta <5!)
• Bugs encontrados pré-produção: +8/mês
• Tempo QA: +2h/sprint (escrever testes) mas -4h/mês (debug produção)
Conclusão: Funcionou! Meta superada! 🎉

A
ACT - 4 semanas

Padronizar:
• Expandir testes E2E para TODOS fluxos críticos (6 semanas)
• Política: "Todo PR precisa ter testes E2E se toca fluxo crítico"
• Treinar TODOS QAs (12 pessoas) em Playwright
• Dashboard: 120 testes E2E, cobertura 85% fluxos

Resultado 6 meses depois:
• Bugs produção total: 45/mês → 12/mês (73% redução!) 🎉
• NPS: 45 → 68
• Time confiante, deploy menos estressante

🎯 Lições Aprendidas

✓ Pareto funcionou: Focamos nos 50% mais importantes
✓ Piloto foi essencial: Aprendemos antes de escalar
✓ Dados guiaram: Não achismos, MÉTRICAS!
✓ Padronização manteve: Virou processo, não evento
