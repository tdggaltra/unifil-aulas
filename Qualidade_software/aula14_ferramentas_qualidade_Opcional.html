📊 7 Ferramentas Clássicas - Parte 1

Fluxograma, Ishikawa e Pareto

Fundamentos Atemporais! 📐

1️⃣ Fluxograma (Flowchart)

Para que serve: Mapear processos, identificar gargalos, documentar workflows

Exemplo em Software:

Processo de Deploy:
1. Developer commit → 2. CI executa testes → 3. Build → 4. Testes E2E → 5. Deploy staging → 6. Aprovação manual → 7. Deploy produção

Quando usar: Onboarding novos devs, otimizar processos, identificar desperdícios

2️⃣ Diagrama de Ishikawa (Espinha de Peixe)

Para que serve: Análise de causa raiz, brainstorming estruturado

6 M's do Ishikawa: Método, Máquina, Mão de obra, Material, Medida, Meio ambiente

Exemplo em Software:

Problema: API lenta (response time >2s)
Causas:
• Método: Queries N+1, sem cache
• Máquina: DB com CPU 90%, sem índices
• Mão de obra: Junior dev não conhece ORM
• Material: Dataset 10M registros sem paginação
• Medida: Sem APM, métricas desatualizadas
• Meio ambiente: Produção compartilha DB com testes

3️⃣ Gráfico de Pareto (80/20)

Para que serve: Priorização, identificar problemas mais impactantes

Princípio: 80% dos problemas vêm de 20% das causas

Exemplo em Software:

Bugs em Produção (último mês):
• Autenticação: 45 bugs (37%)
• Checkout: 32 bugs (26%)
• Dashboard: 18 bugs (15%)
• Relatórios: 12 bugs (10%)
• Outros: 15 bugs (12%)

Ação: Focar nos 2 primeiros = 63% dos bugs resolvidos!

📊 7 Ferramentas Clássicas - Parte 2

Histograma, Dispersão, Controle e Folha de Verificação

4️⃣ Histograma

Para que serve: Visualizar distribuição de dados, identificar padrões

Exemplo: Distribuição de response time API (0-100ms: 15%, 100-200ms: 60%, 200-500ms: 20%, >500ms: 5%)

5️⃣ Diagrama de Dispersão (Scatter Plot)

Para que serve: Identificar correlação entre variáveis

Exemplo: Code coverage vs bugs produção (correlação negativa: +10% coverage = -15% bugs)

6️⃣ Gráfico de Controle (Control Chart)

Para que serve: Monitorar estabilidade processo ao longo do tempo

Componentes: Linha central (média), limites superior/inferior de controle (±3σ)

Exemplo: Build time diário (média 5min, LSC 8min, LIC 2min). Se ultrapassar limites → investigar!

7️⃣ Folha de Verificação (Checklist)

Para que serve: Coleta de dados estruturada, garantir consistência

Exemplo - Code Review Checklist:

☐ Código segue style guide? ☐ Testes unitários presentes? ☐ Coverage ≥80%? ☐ Sem variáveis hardcoded? ☐ Tratamento de erros adequado? ☐ Documentação atualizada? ☐ Performance OK? (Big O análise) ☐ Segurança: input validation, SQL injection ☐ Logs informativos ☐ Commit message descritivo

💡 Combinando Ferramentas

Pareto identifica → Ishikawa analisa → Fluxograma documenta → Controle monitora!
Ferramentas funcionam melhor JUNTAS! 🤝

🔍 Ferramentas de Análise

5 Porquês, 5W2H e FMEA

Pergunte Até Achar a Raiz! 🌳

🔎 5 Porquês (5 Whys)

Origem: Toyota (Taiichi Ohno)

Como usar: Perguntar "Por quê?" 5 vezes até chegar na causa raiz

Exemplo:
1. Problema: Deploy falhou em produção
2. Por quê? Testes E2E falharam
3. Por quê? Ambiente staging diferente de produção
4. Por quê? Variáveis de ambiente não sincronizadas
5. Por quê? Processo manual de atualização
6. Causa Raiz: Falta de automação no sync de configs
Solução: Terraform + CI/CD para infraestrutura como código

📋 5W2H

Para que serve: Plano de ação estruturado

What (O quê?) → Ação a ser executada
Why (Por quê?) → Justificativa
Where (Onde?) → Local
When (Quando?) → Prazo
Who (Quem?) → Responsável
How (Como?) → Método
How much (Quanto?) → Custo

Exemplo:
What: Implementar SonarQube
Why: Melhorar qualidade código, reduzir bugs
Where: CI/CD pipeline
When: Sprint 15 (2 semanas)
Who: Tech Lead João
How: Docker container + integração GitHub
How much: R$0 (open source)

⚠️ FMEA (Failure Mode and Effects Analysis)

Para que serve: Análise preventiva de falhas

Cálculo RPN: Severidade × Ocorrência × Detecção (1-10 cada)

Falha Potencial S O D RPN
DB crash sem backup 10 3 8 240
API key vazada no GitHub 9 5 2 90
CSS quebrado em IE11 2 8 4 64

Ação: Priorizar RPN >100 (backup diário DB, secret scanning CI/CD)

🧪 Ferramentas de Teste

Automatizando a qualidade

Teste Automatizado = Segurança! 🛡️

JUnit (Java)

Testes unitários Java

@Test void testSum() { Calculator c = new Calculator(); assertEquals(5, c.sum(2, 3)); }

Jest (JavaScript)

Testes JS/React/Node

test('adds 1 + 2', () => { expect(sum(1, 2)) .toBe(3); });

Cypress

Testes E2E modernos

cy.visit('/login'); cy.get('#email') .type('user@test.com'); cy.get('#submit').click();

Selenium

Automação browser

Cross-browser, múltiplas linguagens, WebDriver

Postman

Testes de API

Collections, environments, CI/CD integration

Playwright

E2E moderno

Multi-browser, auto-wait, screenshots/videos

🎯 Pirâmide de Testes

🔺 E2E (10%): Cypress, Playwright, Selenium
🔸 Integration (20%): Testes de API, DB
🔻 Unit (70%): JUnit, Jest, PyTest

Regra: Mais testes unitários (rápidos, baratos) e menos E2E (lentos, caros)!

🔬 Análise de Código

Qualidade automatizada

Code Quality = Security + Maintainability! 🔐

🔷 SonarQube

O quê: Plataforma completa de análise de qualidade

Analisa:

• Bugs: Erros de lógica, null pointers
• Vulnerabilidades: SQL injection, XSS, hardcoded secrets
• Code Smells: Código duplicado, funções gigantes, complexidade ciclomática
• Coverage: % código coberto por testes
• Duplicação: % código duplicado

Quality Gate:

✓ Coverage ≥ 80% ✓ Duplicação ≤ 3% ✓ Bugs = 0 ✓ Vulnerabilidades = 0 ✓ Code Smells ≤ 5 ✓ Maintainability Rating ≥ A

Custo: Community Edition (gratuito) ou Enterprise (pago)

📐 ESLint

O quê: Linter JavaScript/TypeScript

Detecta: Erros de sintaxe, variáveis não usadas, padrões inconsistentes

// .eslintrc.json { "extends": "airbnb", "rules": { "no-console": "warn", "indent": ["error", 2] } }

🎨 Prettier

O quê: Code formatter

Formata: JS, TS, CSS, HTML, JSON, Markdown

Benefício: Código consistente, fim das discussões de estilo!

📊 CodeClimate

O quê: Análise de maintainability

Métricas: Maintainability score (A-F), technical debt (horas)

🎯 CodeCov

O quê: Análise de coverage

Features: Coverage diff, badges, integrações

"Qualquer tolo pode escrever código que um computador entenda. Bons programadores escrevem código que humanos entendam." — Martin Fowler

🚀 Ferramentas de CI/CD

Automatizando o pipeline

Deploy Confiante = CI/CD Maduro! 🎯

⚡ GitHub Actions

O quê: CI/CD nativo do GitHub

Vantagens: Gratuito (2000 min/mês), YAML simples, marketplace 10k+ actions

# .github/workflows/ci.yml name: CI Pipeline on: [push, pull_request] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Run tests run: npm test - name: SonarQube Scan uses: sonarsource/sonarqube-scan-action@v2

🔧 Jenkins

O quê: CI/CD open source mais antigo

Vantagens:

  • Flexibilidade total
  • 1500+ plugins
  • Self-hosted (controle total)

Desvantagens:

  • Infraestrutura própria
  • Manutenção trabalhosa
  • UI antiga

🦊 GitLab CI

O quê: CI/CD integrado GitLab

Vantagens:

  • All-in-one (Git + CI/CD)
  • Auto DevOps
  • Kubernetes nativo

YAML: Similar GitHub Actions

🎯 Pipeline Completo (Exemplo)

1️⃣ Commit → Git push
2️⃣ Lint → ESLint + Prettier (30s)
3️⃣ Test → Jest unit + integration (2min)
4️⃣ Build → Docker image (1min)
5️⃣ Scan → SonarQube + Trivy (1min)
6️⃣ E2E → Cypress tests (3min)
7️⃣ Deploy Staging → Kubernetes (2min)
8️⃣ Smoke Tests → Health checks (30s)
9️⃣ Deploy Prod → Blue-green deployment

Tempo total: ~10 minutos commit → produção! 🚀

📊 Ferramentas de Monitoramento

Observabilidade em tempo real

3 Pilares: Métricas + Logs + Traces! 📈

📊 Grafana + Prometheus

O quê: Stack open source de métricas

Prometheus: Time-series database, scraping métricas
Grafana: Dashboards lindos, alertas

Métricas típicas:
• Request rate (req/s)
• Error rate (% errors)
• Response time (p50, p95, p99)
• CPU/Memory/Disk usage
• DB connections pool
• Queue depth

Custo: R$0 (open source)

🐕 Datadog

O quê: APM all-in-one

Features:

  • Métricas + Logs + Traces
  • RUM (Real User Monitoring)
  • Alertas inteligentes
  • Integrações 500+ serviços

Custo: $15/host/mês

🔍 New Relic

O quê: APM tradicional

Features:

  • Transaction tracing
  • Error tracking
  • Distributed tracing
  • Service maps

Custo: Free tier 100GB/mês

📝 ELK Stack (Elasticsearch + Logstash + Kibana)

O quê: Agregação e busca de logs

Elasticsearch: Search engine, armazena logs
Logstash: Coleta e processa logs
Kibana: Visualização e dashboards

Use cases:
• Buscar erros em produção
• Rastrear requests específicos
• Análise de segurança (tentativas login)
• Audit logs

🎯 Observabilidade vs Monitoramento

Monitoramento: "Sistema está lento?" (conhece o problema)
Observabilidade: "POR QUE está lento?" (investiga causa)

Observabilidade = Métricas + Logs + Traces juntos! 🔍

🔄 Workflow Integrado (Do Commit ao Deploy)

1. Developer escreve código
→ Pre-commit hook: ESLint + Prettier formatam

2. Git push → GitHub
→ GitHub Actions CI inicia automaticamente

3. CI Pipeline executa:
→ Lint (ESLint)
→ Unit tests (Jest/JUnit) com coverage
→ Build Docker image
→ SonarQube scan (quality gate!)
→ E2E tests (Cypress)
→ Security scan (Trivy)

4. Quality Gate OK?
→ ✅ SIM: Deploy staging automático
→ ❌ NÃO: Bloqueia merge, notifica dev

5. Staging → Smoke tests
→ Health checks, critical paths

6. Aprovação manual → Deploy Produção
→ Blue-green deployment
→ Grafana monitora métricas
→ ELK coleta logs
→ Alertas se erro rate >1%

Tempo total: 10 minutos! ⚡
